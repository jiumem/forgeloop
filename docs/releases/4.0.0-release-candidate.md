# Forgeloop 4.0.0 发布候选

状态：源码门禁、authenticated Agent evaluation 与本地候选安装回读已通过，并已取得创建 Release PR、合并、推送、Tag 和 GitHub Release 的明确授权。只能从 Release PR 合并后的最终 `main` Commit 发布。

## 发布范围

- Planning Revision 原位更新同一个 Spec，不废弃并复制父 Issue。
- Ticket 完整正文集中在同一草案文档，会话只提供摘要和本轮变化。
- 高风险 Design Reviewer 受父合同权威约束并跨轮复用；第三次连续 non-PASS 后执行尺度审查。
- `grill-with-docs` 判断和维护跨 Ticket 共享的正式 Design Document；`to-spec` 引用，`to-tickets` 按需门禁。
- 每个 Ticket 最多一个普通 Cycle 和一个自动 Correction Cycle，每个 Cycle 最多三轮有效代码或测试修复。
- Cycle 2 只在可信 Exhaustion Diagnosis 后自动进入；耗尽后停止自动修改。
- Standards Reviewer 保持独立职责，Scheduler 只校验和路由诊断。
- `SHARED` Final Integration Gate 成为最后一个实现 Finding 来源，并提供正式 `to-tickets` 修复出口。
- Final Acceptance 收缩为 Scheduler-owned Acceptance Seal，不再创建 Reviewer、Finding 或 repair key。
- 多 Spec Seal、精确原生回读、关闭顺序和恢复语义保持完整。
- 不新增 Event、持久化状态、Tracker Item 类型或 parser。

## 发布命令

只在 Release PR 合并、本地源码与 `origin/main` 一致，并获得单独发布授权后执行：

```bash
release_commit="$(gh api repos/jiumem/forgeloop/commits/main --jq .sha)"
gh release create v4.0.0 \
  --repo jiumem/forgeloop \
  --target "$release_commit" \
  --title "Forgeloop 4.0.0" \
  --notes-file docs/releases/4.0.0-release-notes.md \
  --latest
```

Tag 必须指向 Release PR 合并后的最终 `main` Commit；不得从本地候选提交、未合并分支或旧验证 Head 发布。

## 0. 文档定位

本文不是实现文档，也不是 Codex CLI 参数手册。

> **为什么要有这套方案。**

## 1. 为什么要有这套方案

### 1.1 问题不在“模型不够强”，而在“人肉流程过于机械”

在现实项目中，尤其是连续数天、数周推进的 milestone 开发里，最耗散的环节往往不是编码本身，而是下面这条重复链路：

> 人类读取任务文档
> → 把任务重新组织成 prompt 发给编码 Agent
> → 等待编码结果
> → 再把总结转发给 Review Agent
> → 再把 Review 意见转发回编码 Agent
> → 重复数轮
> → 最后人类再做架构性审查与签字

这条链路的问题，不在于它不能工作，而在于它具有三个结构性缺陷。

**第一，它把最便宜的“搬运劳动”压在人身上。**
任务投喂、结果摘录、意见转发、状态记忆，这些本应是脚本与状态机承担的工作，却被人不断手工完成。

**第二，它没有统一真值源。**
任务边界在方案文档里，执行结果在聊天记录里，审查意见在另一个对话里，当前状态在人的脑子里。系统没有显式状态，因此也没有可复现性。

**第三，它放大了上下文漂移。**
同一个任务，编码 Agent 看到的是一版描述，Review Agent 看到的是另一版摘要，人类理解的又是第三版浓缩。随着轮次增长，边界与目标会持续变形。

因此，这套方案的价值并不在于“让 Codex 写更多代码”，而在于：

> **把连续任务开发中的机械中间层，收敛为一套可执法、可回环、可升级人工裁决的自动工作流。**

---

### 1.2 这不是“万能自动编码器”，而是“连续任务执行框架”

必须明确，本文讨论的对象不是一个普通的 AI 编码助手。

它不以“单次对话生成代码”为目标，而以**连续任务执行**为目标。
它的核心工作方式不是一次生成，而是：

> **任务装配 → 编码执行 → 结构化总结 → 审查裁决 → 自动回环 → 人工签字**

这意味着，这套方案本质上更接近一套**项目内自动执行框架**，而不是一个“更聪明的 prompt 模板”。

也正因为如此，它不是所有开发场景都值得启用。只有当任务具有连续性、可拆分性、可验收性时，它的收益才会明显大于引入成本。

---

### 1.3 这套方案真正解决的，是“连续任务吞吐”问题

如果从第一性原理看，连续开发任务的吞吐取决于三件事：

| 变量          | 传统人工流程        | 自动任务循环方案      |
| ----------- | ------------- | ------------- |
| **任务输入稳定性** | 依赖人转述，容易漂移    | 依赖结构化任务包，边界稳定 |
| **审查回环成本**  | 每轮都要人工搬运      | 自动流转，人工只介入关键点 |
| **状态可见性**   | 状态隐含在人脑和聊天记录中 | 状态显式存在于系统对象中  |

因此，这套方案要优化的不是“单轮生成质量”，而是**多轮执行过程的可持续吞吐**。

## 2. 适用场景与非适用场景

### 2.1 适用场景

这套方案适合以下类型的工作。

#### 场景 A：连续的大任务开发

例如 milestone 开发、阶段性重构、协议迁移、门禁补齐、兼容壳清理、测试与治理链补强。

这些任务的共同特点是：
它们不是一个 prompt 可以一次完成的离散请求，而是一组**相互衔接的工作闭环**。

#### 场景 B：任务已经完成方案化与拆分化

系统适合执行“已经想清楚的任务”，而不适合替代前期产品与架构规划。

也就是说，进入自动循环之前，应当已经具备：

* 方案文档
* 任务拆分文档
* 边界与验收标准
* 相邻任务之间的依赖关系

#### 场景 C：需要 coder-reviewer 多轮回环

当问题不是“生成一个初稿”，而是“修到 reviewer 再也提不出实质性 findings”，自动回环才有明显价值。

#### 场景 D：仍然保留人工最终裁决

这套方案不是无人值守全自动交付系统。
它更适合这样的协作结构：

> 机器负责执行闭环，
> 人类负责架构判断、边界裁决与最终签字。

---

### 2.2 非适用场景

这套方案不适合以下情况。

#### 场景 A：一次性小修小补

例如 10 行左右 bugfix、局部文案修改、临时脚本试验、一次性探索性代码片段。

在这些任务里，引入状态机、任务包、review loop 的成本会大于收益。

#### 场景 B：项目本身缺乏外显规则

如果项目没有明确的编码约束、没有测试入口、没有文档化边界、没有项目级指令文件，那么自动循环只会把混乱复制得更快。

#### 场景 C：任务仍处于需求探索期

如果当前问题还是“到底应该做什么”，而不是“如何稳定推进已拆好的任务”，那应该先做人类规划，而不是启动自动执行框架。

#### 场景 D：高度依赖主观审美判断

例如品牌表达、创意性界面风格、架构品味探索等强主观任务，不应过度自动化。

## 3. 启用前置依赖

这是本文最关键的章节。

**没有前置依赖的满足，就不应启动这套方案。**
原因不是保守，而是这套系统的本质决定了：它依赖外显规则、结构化任务和可复现执行环境。

前置依赖建议分成四类。

---

### 3.1 工具前置依赖

首先，必须具备可用的本地 Codex 运行环境。

这至少包括：

* 已安装并可正常运行的本地 Codex CLI
* 已完成认证配置，能够稳定执行 `codex exec`
* 本地目标项目目录可被 Codex 访问
* 当前环境允许非交互运行
* 后续能够支持结构化结果输出与文件落盘

这里要特别强调一点：

> 这套方案并不是“聊天驱动”的。
> 它依赖的是 **CLI 驱动的非交互执行能力**。

因此，若本地 Codex 仍停留在手工对话阶段，或者 CLI 运行不稳定，则不应进入自动循环建设。

---

### 3.2 项目前置依赖

其次，项目本身必须已经“规则外显”。

最低要求有三项：

#### 第一，项目内已有 `AGENTS.md` 或等价项目级指令文件

这份文件的作用不是写产品愿景，而是为 Codex 提供**持续一致的项目级编码约束**。

它至少应覆盖：

* 项目边界
* 编码纪律
* 验证入口
* 禁止事项
* 必须遵守的工程契约

#### 第二，项目具备可执行的验证命令

至少应存在：

* 构建命令
* 测试命令
* lint / gate / 检查脚本中的一类或多类

否则 coder 虽然能提交代码，但系统无法形成“完成-验证-审查”的闭环。

#### 第三，项目不是纯口头治理

如果项目当前仍然依赖“开发者彼此默认知道”的隐性规则，这套方案会迅速失稳。

> 自动化系统不怕规则严格，最怕规则只存在于人脑中。

---

### 3.3 任务前置依赖

这是你刚才指出的重点，我完全认同，而且必须写透。

这套方案适合的是**连续的大任务开发**，而不是零散临时请求。
因此，在任务层面，至少应具备以下条件。

#### 第一，已经存在 milestone 或连续任务序列

也就是说，系统不是拿来处理一条孤立需求，而是拿来推进一组有先后关系的任务。

#### 第二，已经有方案文档

方案文档负责回答“为什么做、总体结构是什么、边界在哪”。

#### 第三，已经有任务拆分文档

任务拆分文档负责把大目标切成一组可独立验收的 work item。

#### 第四，任务之间存在明确依赖与完成定义

这意味着至少应能回答：

* 当前任务的 must-do 是什么
* 明确不做什么
* done criteria 是什么
* 前置任务是否已完成
* 后续任务何时可接续推进

换句话说：

> **没有任务拆分，就没有自动编排。**
> **没有完成定义，就没有自动审查。**

---

### 3.4 组织前置依赖

最后，团队层面也必须具备基本条件。

#### 第一，存在明确的人类责任人

必须有人承担以下职责：

* 方案审批
* 边界裁决
* 最终人工 review
* 合入决定

自动循环不是取消责任，而是把责任从“搬运劳动”迁移到“关键裁决”。

#### 第二，团队接受结构化结果优先

也就是说，大家要接受：

* 任务状态以结构化对象为真值
* 自然语言说明是解释层，不是执法层
* 进入下一任务不是靠口头说“差不多了”，而是靠状态机与规则推进

#### 第三，团队接受阶段性通知而非轮次骚扰

通知机制应当围绕关键事件设计，而不是每一轮 coder-reviewer 都打断人。

## 4. 用户如何使用这套方案

从采用角度看，用户并不是直接“启动 AI”，而是在启动一条**带前提、带状态、带回环的任务链**。

推荐采用流程如下。

### 第一步：准备本地 Codex 环境

确认 CLI 已可执行，认证正常，目标项目目录可访问。

### 第二步：确认项目规则已外显

检查项目中是否已经存在 `AGENTS.md`、验证命令、基本工程约束。

### 第三步：准备方案文档与任务拆分文档

确认当前工作不是模糊意向，而是已被方案化、拆分化的连续任务。

### 第四步：为任务生成结构化任务输入

无论最终具体形式是什么，至少需要让系统获得以下信息：

* 当前任务目标
* 必做项
* 明确不做项
* 验收标准
* 前置依赖
* 必跑检查

### 第五步：启动自动任务循环

由 controller 负责任务装配、调用 coder、调用 reviewer、推进状态。

### 第六步：在人类介入点完成人工裁决

当出现以下事件时，由人类介入：

* 法理或边界冲突
* 自动回环达到 clean 状态后的人工审查
* 阶段性里程碑完成后的确认

### 第七步：进入下一个任务

在当前任务完成并通过人工确认后，系统再根据规则与建议推进下一任务。

## 5. 这套方案解决什么，不解决什么

### 5.1 它解决的问题

这套方案主要解决以下问题：

#### 一、减少机械搬运劳动

把“任务转述、结果转发、状态记录、轮次推进”从人身上剥离出来。

#### 二、降低上下文漂移

通过结构化任务对象与审查对象，减少多轮对话中的边界变形。

#### 三、提升连续任务吞吐

让 milestone 级任务不再依赖人肉逐轮串联。

#### 四、保留人类在关键点的控制权

它不是全自动替代，而是自动执行 + 人类签字。

---

### 5.2 它不解决的问题

这套方案不负责以下内容：

#### 一、替代前期产品与架构规划

如果方案尚未成形，自动循环无法替你思考目标。

#### 二、替代人工架构品味判断

系统可以收敛 correctness，但不能替代人对整体架构优雅性的最终判断。

#### 三、替代项目规则建设

若项目没有 `AGENTS.md`、没有验证入口、没有任务边界，问题不在自动化，而在项目治理本身。

#### 四、把任何开发工作都自动化

它是针对连续、可拆、可验收任务的专用框架，不是无边界万能工具。

## 7. 推荐采用路径

从产品角度，这套方案不应一上来就覆盖所有项目，而应采用分阶段推进。

### 阶段一：单 milestone 试点

选择一个边界清晰、任务已拆分、验证入口稳定的连续任务，先跑通最小闭环。

### 阶段二：连续任务扩展

在首个 milestone 成功后，再扩展到同类连续任务。

### 阶段三：接入更完整的控制与观测

当基础回环稳定后，再补：

* 更完整的 mock
* 更好的状态可视化
* 更多通知与汇总能力
* 更复杂的任务调度策略

这个顺序很重要。
因为这套系统最怕的不是“能力不够强”，而是“平台先行、采用滞后”。

---

## 8. 总结裁决

这套方案成立的前提，不是“Codex 足够强”，而是下面三件事同时成立：

> **项目规则已外显。**
> **任务已经方案化与拆分化。**
> **团队愿意把机械执行交给系统，把关键裁决保留给人。**

只有在这三个条件同时成立时，自动任务编码循环才不是噱头，而会成为真正有价值的开发基础设施。

再压缩成一句话：

> **这套方案适合连续任务的自动执行，不适合无边界问题的自动想象。**

# Tracker Runtime Routing

Read `Tracker Runtime Operations`, generated by `$setup-forgeloop` in `docs/agents/issue-tracker.md`, before acting. It defines commands and platform capabilities. This Reference defines only shared semantic boundaries.

## GitHub

Use `gh` for all operations. Express relationships with native Sub-issues and Dependencies when available, otherwise use only the configured body fallback.

Place the competitive `RUN_CLAIMED` Comment on the Spec or Initiative root. The earliest valid server-side root Claim wins. After winning, claim the current Ticket through its native assignee state. Do not duplicate that Ticket Claim as an Event. Use GitHub facts for Frontier, Branch, Commit, PR, Required Checks, merge, and closure.

Append minimal idempotent checkpoints as Issue Comments. Hold the two Ticket Reviewer results until both finish, then write one combined `REVIEW_RESULT`. A closed Ticket with valid Integration makes its assignee Claim inactive. In a multi-Spec Run, keep member Specs Open through Initiative Acceptance, then close members first and the parent last on Initiative `PASS`. A closed root with final Acceptance makes its historical root Claim inactive. Do not delete historical Claim Comments. On authentication, Issue/PR permission, protection, or checks failure, keep Items Open and report a diagnosable error; never fall back to Local.

## GitLab

Use `glab` for all operations. Prefer native Blocking where available and otherwise use only the configured fallback relationships.

Place the competitive `RUN_CLAIMED` Note on the Spec or Initiative root. The earliest valid server-side root Claim wins. After winning, claim the current Ticket through its native assignee state. Do not duplicate that Ticket Claim as an Event. Use GitLab facts for Frontier, Branch, Commit, MR, pipelines, merge, and closure.

Append minimal idempotent checkpoints as Issue Notes, with the two Ticket Verdicts combined only after independent collection. A closed Ticket with valid Integration makes its assignee Claim inactive. In a multi-Spec Run, keep member Specs Open through Initiative Acceptance, then close members first and the parent last on Initiative `PASS`. A closed root with final Acceptance makes its historical root Claim inactive. Do not delete historical Claim Notes. On authentication, project permission, protection, or pipeline failure, keep Items Open and report a diagnosable error; never fall back to Local.

## Local Markdown

Keep the configured Tracker under `.scratch/<feature>/`. Compete for the root Scheduler by atomically creating `scheduler.lock`, then claim only the current Ticket with its configured atomic Ticket lock. Store `run_id` in each owned lock. The Scheduler lock and Ticket lock are native Claim facts; do not duplicate the Ticket lock as an Event.

Append minimal idempotent checkpoints under the configured Agent Run section. Stop on file conflicts, invalid references, an unowned dirty worktree, or inconsistent Git facts. Represent candidate and integration facts through Git Branches and Commits. After writing a Ticket `resolved`, atomically remove only its owned Ticket lock. In a multi-Spec Run, keep member Specs Open through Initiative Acceptance, then resolve members first and the parent last on Initiative `PASS`. After root closure, atomically remove only the owned `scheduler.lock`. Never release either lock before its completion gate or when the stored Run ID differs.

## Platform-Equivalent Result

All runtimes must produce the same domain outcome: one root Scheduler Claim, at most one native active Ticket Claim, four Coder results, two independently collected Verdicts persisted together, at most two ordinary repair rounds, explicit Integration, fresh layered Acceptance, explicit pause/cancel behavior, and recovery from Tracker and Git without old child threads. CLI commands and physical storage must not change completion semantics.

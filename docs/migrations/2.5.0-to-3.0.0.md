# Forgeloop 2.5.0 → 3.0.0 迁移指南

## 变更摘要

`3.0.0` 从本地 Initiative 文档与 Milestone 账本切换为 Tracker 驱动的 Spec/Ticket 模型。这是破坏性变更：`grill-initiative`、`plan-initiative`、`run-initiative-sequences` 被移除；`run-initiative` 不再读取或写入 `DESIGN.md`、`PLAN.md`、`LEDGER.md`。

替代关系：

- `grill-initiative` → `$grill-with-docs`，大型探索使用 `$wayfinder`；
- `plan-initiative` → `$to-spec` + `$to-tickets`；
- `run-initiative-sequences` → `$run-initiative` 的多 Spec Initiative；
- 旧 `run-initiative` → 从正式 Spec/Tickets 运行的新 `$run-initiative`。

## 历史产物

`docs/initiatives/completed/`、`archived/`、`handoff/` 与 `recommendations/` 原地只读保留。新版 Setup 只检测并报告，不移动、不删除、不自动转换，也不把这些文件当作实时状态。

## 活动 Initiative 的两条路径

### 路径 A：固定 2.5.0 完成

适合已执行较深、继续成本低于迁移成本的工作：

1. 从源码 Commit `a32f7a5d1eceace7215321dd7168dc7d07dac249`（Tag `v2.5.0`）恢复旧插件。
2. 在隔离分支/工作树中按旧流程完成。
3. 不让 `3.0.0` 同时修改同一活动 Initiative。

### 路径 B：重新生成 Tracker 工作

适合尚未执行或希望切换到新门禁的工作：

1. 运行 `$setup-forgeloop`，配置 GitHub、GitLab 或 Local Markdown 与 Integration Policy。
2. 逐个 Initiative 预览旧设计、未完成范围和已完成历史；用户明确确认迁移范围。
3. 用 `$to-spec` 重新发布正式 Spec；不要机械复制旧 Milestones。
4. 用 `$to-tickets` 从确认后的 Spec 重新拆垂直 Tickets 与 Blocking Graph。
5. 已完成 Milestone 只作为历史证据，不创建伪完成 Ticket。
6. 原文件保持原位；迁移重跑按 Tracker 引用识别，避免重复创建。

未执行的 `DESIGN.md` 可以在用户确认后成为 Spec 输入；未执行的 `PLAN.md` 只能在确认 Spec 后重新拆 Tickets；`LEDGER.md` 不迁移为新运行状态。

## 回退

若迁移预览或新 Tracker 配置不满足要求，停止且不发布新 Items，继续保留原文件。已经创建但未执行的新 Spec/Tickets 应明确标为取消或关闭原因，而不是删除历史；随后可回到路径 A。不得让两套流程同时写同一工作范围。

## 发布边界

本指南不授权推送、PR、Tag 或 Release。`2.5.0` Tag/Release 与 `3.0.0` 发布命令只在单独取得发布授权后执行。

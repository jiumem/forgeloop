# Tracker Runtime Routing

Read `Tracker Runtime Operations`, generated by `$setup-forgeloop` in `docs/agents/issue-tracker.md`, before acting. It defines commands and platform capabilities. This Reference defines only shared semantic boundaries.

## GitHub

Use `gh` for all operations. Express relationships with native Sub-issues and Dependencies when available, otherwise use only the configured body fallback.

Place the competitive `RUN_CLAIMED` Comment on the Spec or Initiative root. The earliest valid server-side root Claim wins. After winning, claim the current Ticket through its native assignee state. Do not duplicate that Ticket Claim as an Event. Use GitHub facts for Frontier, Branch, Commit, PR, Required Checks, merge, and closure.

Append minimal idempotent checkpoints as Issue Comments. Publish the Prepared Literal Payload with `gh issue comment --body-file <payload-file>` or the equivalent `--body-file -` stdin channel, never as a dynamic body argument. Preserve the returned Comment URL as a native reference, then complete the Native envelope through an API GET that returns the Comment ID, server timestamp, and raw body. Confirm that body before advancing. Hold the two Ticket Reviewer results until both finish, then write one combined `REVIEW_RESULT`. A closed Ticket with valid Integration makes its assignee Claim inactive. In a multi-Spec Run, keep member Specs Open through Initiative Acceptance, then close members first and the parent last on Initiative `PASS`. A closed root with final Acceptance makes its historical root Claim inactive. Do not delete historical Claim Comments. On authentication, Issue/PR permission, protection, or checks failure, keep Items Open and report a diagnosable error; never fall back to Local.

## GitLab

Use `glab` for all operations. Prefer native Blocking where available and otherwise use only the configured fallback relationships.

Place the competitive `RUN_CLAIMED` Note on the Spec or Initiative root. The earliest valid server-side root Claim wins. After winning, claim the current Ticket through its native assignee state. Do not duplicate that Ticket Claim as an Event. Use GitLab facts for Frontier, Branch, Commit, MR, pipelines, merge, and closure.

Append minimal idempotent checkpoints as Issue Notes. Use `glab api` with a file-backed field such as `--field body=@<payload-file>`; do not pass dynamic Payload text to the string-only Note shortcut, and specifically not `glab issue note --message`. `--input` accepts a safely generated complete JSON request body, not bare Markdown. Preserve the created Note ID and read it back for the native Note reference, server timestamp, and body. Do not submit `created_at`; accept the platform value. Confirm the body before advancing. Keep the two Ticket Verdicts combined only after independent collection. A closed Ticket with valid Integration makes its assignee Claim inactive. In a multi-Spec Run, keep member Specs Open through Initiative Acceptance, then close members first and the parent last on Initiative `PASS`. A closed root with final Acceptance makes its historical root Claim inactive. Do not delete historical Claim Notes. On authentication, project permission, protection, or pipeline failure, keep Items Open and report a diagnosable error; never fall back to Local.

## Local Markdown

Keep the configured Tracker under `.scratch/<feature>/`. Compete for the root Scheduler by atomically creating `scheduler.lock`, then claim only the current Ticket with its configured atomic Ticket lock. Store `run_id` in each owned lock. The Scheduler lock and Ticket lock are native Claim facts; do not duplicate the Ticket lock as an Event.

Append the same Prepared Literal Payload under the configured Agent Run section with a non-shell file operation, preserving the existing atomic append and ownership rules. Capture its record position and append timestamp, then read back that exact record and confirm its body before advancing. Stop on file conflicts, invalid references, an unowned dirty worktree, or inconsistent Git facts. Represent candidate and integration facts through Git Branches and Commits. After writing a Ticket `resolved`, atomically remove only its owned Ticket lock. In a multi-Spec Run, keep member Specs Open through Initiative Acceptance, then resolve members first and the parent last on Initiative `PASS`. After root closure, atomically remove only the owned `scheduler.lock`. Never release either lock before its completion gate or when the stored Run ID differs.

## Platform-Equivalent Result

The Repair Loop is the authoritative repair-budget contract; this runtime summary must mirror it.

All runtimes must produce the same domain outcome: one root Scheduler Claim, at most one native active Ticket Claim, one initial Coder result plus at most three ordinary repair results per repair cycle, two independently collected Verdicts persisted together, explicit Integration, fresh layered Acceptance, explicit pause/cancel behavior, and recovery from Tracker and Git without old child threads. Automatic renewal keeps the same Ticket, Run, and Branch and opens no replacement Ticket. CLI commands and physical storage must not change completion semantics.

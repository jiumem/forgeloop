# Forgeloop 3.6.0

Forgeloop 3.6.0 把 TDD 从可选的实现建议接入 `run-initiative` 的正式 Coder 合同：当 Ticket 要求可复现的行为变化时，Agent 必须先从已批准的公共 Seam 观察真实 Red，再以同一条仓库根目录命令取得 Green；其他验证类型保留各自正确的证据语义。

## 按验证策略调用 TDD

- Scheduler 交给 Coder 的 Role Task Pack 现在必须包含已批准的 Validation Entries、公共 Seam、Acceptance Prerequisites 和证据合同；
- 可复现的新增行为、缺陷修复或恢复行为必须在修改生产代码前调用 `$tdd`；Role Task Pack 中已批准的 Seam 直接作为授权，不在子任务中重复向用户确认；
- Red 必须由目标行为尚未实现导致，不能来自语法错误、无关服务不可用或刻意破坏的 fixture；
- 当 Base 尚无测试时，允许把新测试应用到冻结的 Base 生产代码形成 test-only Red；生产实现仍保持 Base 状态；
- Green 必须在最终固定 Head 上运行同一条仓库根目录公共命令，防止用内部 helper、替代入口或不同测试掩盖未闭合的行为；
- 后续获得授权的代码修复重新进入相同策略；只读 Repair Diagnosis 与 Exhaustion Diagnosis 不调用 `$tdd`。

## 不伪造 Red

- 行为保持型变更不制造人为失败，而是在 Base 和 Head 上验证相同公共行为，并提交旧路径退出、事实源收敛或兼容性完成等可定位结构证据；
- 声明依赖外部条件的验证只能使用批准的真实观察路径；当前环境无法合法建立条件时返回 `IMPLEMENTATION_BLOCKED`，不能用 mock、recording adapter 或 harness 生成的结论替代；
- `NO_CHANGE_REQUIRED` 继续要求 `Base == Head`、空 Diff 和现有行为的可观察证据，不被强行套入 Red/Green。

## 事实权威与职责边界

- Scheduler 只确认 Coder 是否按已批准策略提交了完整、无矛盾且绑定到正确 Base、test-only Red、Candidate 或 Head 的证据；它不重跑完整验证，也不判断证据是否可信；
- Spec Reviewer 与 Standards Reviewer 独立审查同一固定 Head，并根据批准的公共 Seam 判断事实来源和证明力度；
- harness 和 adapter 可以传递 challenge、启动产品或读取 opaque result，但不能签发测试随后接受的 `ready`、`reconciled`、`authorized` 等产品结论；
- mock-only 成功、caller-shaped evidence、内部 helper 或根据请求拼装预期 evidence 的 recording adapter，无法证明合同要求的真实产品行为时必须形成 Blocking Finding。

## 兼容性

- 3.6.0 保持 20 个正式 Skill、11 个用户入口和 9 个模型可调用能力不变；
- 不新增运行时状态、结果枚举、parser 字段、Tracker 状态或配置迁移；
- TDD overlay 继续由上游映射生成，避免本地 Skill 与同步来源形成两份事实；
- 新规则只约束后续 Coder 执行和 Review，不自动改写既有 Spec、Ticket 或历史证据。

## 验证

- Runtime Contract 提示词契约测试覆盖三类验证策略、test-only Red、修复阶段调用边界和产品事实来源；
- authenticated Codex Agent 语义冒烟验证覆盖可复现行为、行为保持、外部条件、`NO_CHANGE_REQUIRED`、Scheduler 边界和 Reviewer 反自证场景；
- Release Validator、Runtime Contract、Skill UI 元数据、上游同步和 Plugin Validator；
- Python 全量单元测试与本地 Marketplace 候选重装；
- Release Tag、GitHub Release 与最终 `main` Commit 精确绑定。

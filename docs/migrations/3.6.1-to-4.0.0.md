# Forgeloop 3.6.1 → 4.0.0 迁移指南

4.0.0 不迁移 Tracker Schema，也不改写历史 Spec、Ticket、Run 或 Checkpoint。需要迁移的是执行合同：Planning Revision、修复周期、Final Integration Gate 和 Final Acceptance 的职责边界发生了不兼容变化。

## 升级前

1. 找出仍持有 `RUN_CLAIMED`、Ticket Claim 或活动子任务的 3.6.1 Run。
2. 不要在同一个活动 Run 内切换到 4.0.0。优先使用原版本完成它；如果必须放弃，先按 3.6.1 合同明确取消并释放其 Claims，再用 4.0.0 创建新 Run。
3. 保留所有历史事件和原生 Tracker 事实，不修改旧 `ACCEPTANCE_RESULT`、`RUN_PAUSED` 或 repair key。
4. 安装 4.0.0 后新建 Codex 任务，避免旧任务继续使用缓存中的 3.6.1 Skill 文本。

## 已有 Spec 与 Ticket

- 已发布 Spec 和 Ticket 继续有效，不需要批量补写 Formal Design Document。
- 后续如果 `to-tickets` 发现多个 Ticket 必须共享仍未明确的实现决定，它会返回 `DESIGN_DOCUMENT_REQUIRED`；显式调用 `grill-with-docs` 创建或原位修订文档，再重新运行 `to-tickets`。
- 需要修改父 Spec 时，使用 `to-spec` 的 Planning Revision 原位更新同一个 Issue。不要关闭旧 Spec 再创建替代 Spec。
- 已经存在的修复 Ticket 继续作为普通 Ticket 执行；4.0.0 不会因为历史 repair key 再创建第二张 Ticket。

## 活动修复周期

- 4.0.0 的每个 Ticket 最多只有 Cycle 1 和唯一 Cycle 2，每个 Cycle 最多三轮有效代码或测试修复。
- 3.6.1 Run 如果已经完成多次自动续配，不得把历史轮次重新编号后继续套用 4.0.0 预算；应使用原版本完成，或者明确取消后重新规划。
- Cycle 1 耗尽后不询问用户是否继续。只有 Correction Coder 给出可信 `AUTO_REPAIR_RENEWAL` 且恢复记录精确回读成功时，才自动进入 Cycle 2。
- Cycle 2 耗尽后保持 `IMPLEMENTATION_BLOCKED` 或进入真实 `CONTRACT_BLOCKER`，不会创建 Cycle 3。

## Acceptance Repair

- 4.0.0 不再从 Final Acceptance 产生 Finding、repair key 或修复 Ticket。
- 3.6.1 中已经确认的 Acceptance Seal 和历史 Acceptance 事件保持原样，不需要转换。
- 仍暂停在旧 `PAUSED_FOR_ACCEPTANCE_REPAIR` 路径的 Run 不应由 4.0.0 直接重新解释。使用 3.6.1 完成既有修复，或者明确取消旧 Run，再让 4.0.0 从当前 Tracker 与 Git 事实建立新运行。
- 新的实现缺口最迟由 `SHARED` Spec 的 Final Integration Gate 发现，并通过正式 `RUN_PAUSED reason=FINAL_GATE_FINDING` 交给 `to-tickets`。Acceptance Seal 阶段只处理缺失、过期或矛盾的既有事实。

## 升级验证

在仓库根目录运行：

```bash
python3 tooling/forgeloop/scripts/validate_suite.py \
  --mode release \
  --plugin-root plugins/forgeloop

python3 -m unittest discover \
  -s tooling/forgeloop/tests \
  -p 'test_*.py'
```

确认 Manifest 为严格版本 `4.0.0`，安装缓存只包含 `.codex-plugin` 与 `skills`，并在新 Codex 任务中确认插件已启用。

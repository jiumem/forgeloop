# Forgeloop 3.5.0 发布候选

状态：发布前门禁已通过并取得用户发布授权；仅允许从 Release PR 合并后的最终 `main` Commit 发布。

## 发布范围

- 在 `to-spec` 首次 Tracker 写入前，以完整候选 Spec、已解决上下文、代码事实和相关 ADR 执行语义化 Necessity Review。
- 从当前阻塞和失败机制出发，寻找最小权威生产 Seam，优先复用现有事实、owner、接口和生命周期。
- 每个新概念绑定当前失败证据；未来用途、未来专用基础设施和无当前消费者 helper 不能进入正式 Spec。
- 在既有授权内删除无证据复杂度；候选变化后从已批准测试 Seam 起重跑全部发布门，禁止复用旧判断。
- 更小方案需要新的产品、Scope、失败语义、ADR 或公开接口决定时返回 `CONTEXT_INSUFFICIENT`。
- 旧机制退出 Proof 以受影响生产路径为边界，仅在正式全局淘汰时要求合法生产调用为零。
- `to-tickets` 只执行薄层拆分最小性：默认一至三个完整纵向 Ticket；超过三个时，每个额外 Ticket 必须提供当前证据。
- enabling change 与其当前消费者合并；prefactoring 默认留在首个消费它的纵向 Ticket。只有独立形成当前可观察系统保证时才可单独成票。
- 拆票若要求改变批准契约，返回 `CONTRACT_BLOCKER`，不在下游重新发明 Spec。
- 澄清 `spec-standards-review` 的模型调用边界：只有仓库标准、没有 originating Spec 时仍可加载；探索、影响分析和调试继续排除。
- 20 个 Skill、11 个用户入口和 9 个模型可调用能力保持不变。

## 发布命令

只在 Release PR 合并且本地 `main` 与 `origin/main` 同步后执行：

```bash
release_commit="$(git rev-parse origin/main)"
gh release create v3.5.0 \
  --repo jiumem/forgeloop \
  --target "$release_commit" \
  --title "Forgeloop 3.5.0" \
  --notes-file docs/releases/3.5.0-release-notes.md \
  --latest
```

Tag 必须指向合并后的最终 `main` Commit；不得从本地候选提交或未合并分支发布。

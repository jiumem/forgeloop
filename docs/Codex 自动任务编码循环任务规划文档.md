## 0. 文档定位

如果说《采用说明》回答的是“值不值得做、何时该做”，《设计方案》回答的是“系统应该如何设计”，那么本文回答的是另一类问题：

> **这套系统首版究竟要按什么顺序落地，以及每一步做到什么程度才算成立。**

因此，本文不是原则宣言，也不是实现细节手册，而是**实施规划文档**。

它站在系统建设视角，解决四件事：

* 给首版范围封板，防止规划阶段平台化失控
* 把系统拆成若干可独立验收的 milestone 与 work item
* 明确各阶段的输入、输出、依赖与退出条件
* 让后续 task packet 的生成有稳定上位依据

本文与现有文档的关系如下：

| 文档       | 主要回答什么                       |
| -------- | ---------------------------- |
| **采用说明** | 为什么做、何时启用、前置依赖是否满足           |
| **设计方案** | 结构化合同、状态机、角色边界、接口与默认配置       |
| **任务规划** | 首版做什么、按什么顺序推进、拆成哪些任务、如何验收成功 |

## 1. 规划目标与规划边界

### 1.1 规划目标

首版规划目标只有一个：

> **在单一仓库、单一 milestone、单任务串行调度前提下，跑通一条可重复的自动编码-审查-回环-人工签字闭环。**

围绕这个总目标，本文定义五个具体交付目标：

| 目标           | 解释                                  |
| ------------ | ----------------------------------- |
| **可运行**      | 系统能从 `task_packet` 启动，并实际驱动 coder 与 reviewer |
| **可裁决**      | controller 能基于结构化结果推进状态，而不是靠自然语言猜测 |
| **可留痕**      | 每轮运行、状态跃迁、最终结论都有对象化产物可追溯            |
| **可中断**      | 遇到冲突、回环超限、依赖缺失时能稳定升级人工，而不是死循环       |
| **可试点**      | 能在一个真实连续任务链中小范围试跑，而不是停留在 mock 演示   |

### 1.2 规划边界

首版不是完整平台，只是一套**最小成立系统**。

因此，规划必须主动收窄边界：

* 只支持**单仓库**
* 只支持**单任务串行推进**
* 只支持**单个 controller 进程**
* 只支持**coder / reviewer 两个执行角色**
* 只要求**本地 CLI 驱动**
* 只要求**文件级状态持久化**
* 只做**关键节点通知**

凡是超出上述边界的内容，都不应进入首版关键路径。

## 2. 首版开发方法裁决

### 2.1 采用“合同优先的 TDD”

对于这个项目，首版最合适的开发方式不是“先把功能拼起来再补测试”，而是：

> **先定义合同，再设计 mock 剧本，再用测试驱动 controller 与 adapter 落地。**

原因不是教条，而是项目性质决定的。

这个系统真正的失败点通常不在“Codex 命令有没有成功调用”，而在：

* 状态是否被错误推进
* `review_result` 是否被错误解释
* 回环何时应继续、何时应停止
* 哪些字段是业务真值，哪些字段只是解释材料

因此，首版开发必须把重心放在**结构化合同与裁决规则**，而不是把 CLI 调用尽快跑起来。

### 2.2 开发顺序裁决

首版推荐顺序如下：

1. 先定义四个数据模型 schema。
2. 再定义状态机与 controller 裁决规则。
3. 然后定义内外两层接口的合同。
4. 接着设计 mock 剧本与 fixture。
5. 先写 controller 和 adapter 的测试。
6. 再写实现代码。
7. 最后补少量真实 CLI 集成 smoke test。

这意味着：

* schema 先于实现
* mock 先于大部分业务代码
* controller 测试先于 controller 实现
* 真实 Codex 集成只做收口验证，不承担主要正确性证明

### 2.3 测试重心裁决

首版测试重心应放在以下四类对象：

| 测试对象 | 重点 |
| --- | --- |
| **schema 校验** | 四个数据模型的必填字段、枚举值、结构边界 |
| **状态机测试** | `advance_state()` 在不同输入下的跃迁是否正确 |
| **接口合同测试** | adapter 输入输出是否满足结构化约束 |
| **集成 smoke test** | 真实 `codex exec` 链路是否至少可跑通最小闭环 |

### 2.4 不推荐的开发顺序

以下顺序会显著增加返工风险：

* 先写 CLI 调用细节，再回头补 schema
* 先把 coder / reviewer prompt 跑起来，再定义 controller 裁决规则
* 先做真实集成，再补 mock fixture
* 把 JSONL 事件流当作主要业务真值来测试

## 3. 首版范围封板

### 3.1 首版必须交付的能力

v1 必须交付以下八类能力。

#### 一、任务真值输入

系统必须能够接收并校验 `task_packet`，至少覆盖：

* 任务身份
* must-do
* must-not-do
* done criteria
* required checks
* depends_on

#### 二、运行态状态机

系统必须实现设计文档中的最小状态流：

```text
NEW -> CODING -> REVIEWING -> NEEDS_FIX -> CODING
REVIEWING -> REVIEW_CLEAN -> HUMAN_REVIEW -> DONE
ANY -> NEEDS_HUMAN_RULING
ANY -> BLOCKED
```

#### 三、Codex CLI 适配层

系统必须通过统一 adapter 调用 `codex exec`，而不是把命令拼接散落在业务代码里。

#### 四、结构化输出约束

coder 与 reviewer 的最终结果，必须通过 schema 约束落盘为结构化 JSON。

#### 五、deterministic controller

controller 必须按照显式规则推进状态，不自由重判代码质量或任务边界。

#### 六、回环与停止条件

至少要支持：

* blocking findings 回 coder
* checks 未过回 coder
* must-do 未完成回 coder
* reviewer 无法可靠判断时升级人工
* 回环超限时升级人工

#### 七、人工签字入口

系统必须保留人工批准入口，用于 `HUMAN_REVIEW -> DONE` 的正式跃迁。

#### 八、最小 mock 与测试

至少要覆盖 controller 核心裁决，而不是只验证命令有没有跑起来。

### 3.2 首版明确不做的内容

以下内容不是不能做，而是**不进入 v1 的必做清单**：

* 多任务并行调度
* 自动任务拆分生成
* dashboard 与可视化工作台
* GitHub Action / CI 深整合
* reviewer 内部多 subagent 并行审查
* 智能优先级排产
* 跨仓库依赖编排
* 长期运行的队列与 worker 服务化
* 基于 JSONL 事件流的复杂实时观测面板

### 3.3 首版成功定义

只要满足以下条件，就可以认定 v1 成立：

1. 能用一个真实 `task_packet` 启动任务。
2. coder 与 reviewer 都能输出满足 schema 的结构化结果。
3. controller 能在 `clean`、`needs_fix`、`needs_human_ruling` 三类结果下稳定推进。
4. 系统能在至少一个真实任务链中完成一次 `NEW -> DONE`。
5. 整个过程中不依赖人工搬运中间文本作为状态真值。

## 4. 实施阶段划分

为避免“大设计先行、实现迟迟不落地”，首版建议拆成五个阶段。

| 阶段 | 名称              | 目标                           |
| -- | --------------- | ---------------------------- |
| **M0** | 规划封板与仓库脚手架      | 建立目录、规则、基础文档与样例输入           |
| **M1** | 核心合同与状态机        | 先把四个模型、状态规则与接口合同立住          |
| **M2** | Mock 与测试基线       | 先写 fixture 与核心裁决测试           |
| **M3** | 实现单轮闭环与自动回环     | 在测试约束下实现 adapter、controller 与回环 |
| **M4** | 真实集成与试点收口       | 用真实 Codex 运行做 smoke test 和试点 |

这五个阶段不是平行菜单，而是严格前后依赖。

## 5. Milestone 任务拆分

### 5.1 M0：规划封板与仓库脚手架

M0 的目标不是写业务代码，而是给后续任务提供稳定地基。

#### M0-1 建立仓库约束与目录骨架

**目标：**
建立首版推荐目录，让规则、schema、adapter、state、mock 有固定落点。

**产出：**

* `AGENTS.md`
* `.codex/prompts/`
* `automation/`
* `schemas/`
* `runs/`
* `mock/`

**完成标准：**

* 目录结构已创建
* 每个目录的职责已写明
* 后续文档与实现都按该目录收敛

#### M0-2 确定首版运行方式与持久化策略

**目标：**
封板 v1 的运行形态，避免后续反复摇摆。

**裁决建议：**

* 运行方式：本地命令行触发
* 持久化方式：文件落盘优先
* 调度方式：单任务串行
* 通知方式：先做最小事件通知

**完成标准：**

* 运行入口明确
* 状态落盘位置明确
* 不再讨论服务化、多 worker、分布式队列

#### M0-3 准备样例任务输入

**目标：**
提供至少一个真实感足够强的 `task_packet` 示例，为后续开发与 mock 共用。

**完成标准：**

* 至少有 1 个 `task_packet` 样例
* 样例具备 must-do / must-not-do / done criteria / checks / depends_on
* 可以直接用于 M2 联调

### 5.2 M1：核心合同与状态机

M1 是首版系统的法律层。

#### M1-1 定义四个 schema

**目标：**
为 `task_packet`、`coder_result`、`review_result`、`task_state` 建立稳定结构。

**完成标准：**

* 四个 schema 已存在
* 必填字段与可选字段有明确界限
* 能支持 controller 自动裁决所需最小信息

#### M1-2 实现状态枚举与跃迁规则

**目标：**
把设计文档中的状态图写成明确规则，而不是散落在 prompt 和自然语言里。

**完成标准：**

* 状态枚举可程序化引用
* 状态跃迁表可测试
* 关键异常出口已编码

#### M1-3 实现 task_state 读写

**目标：**
让 controller 有稳定的运行台账。

**完成标准：**

* 可创建初始 `task_state`
* 可绑定轮次结果
* 可记录 round_no、current_status、next_action、closure_summary

#### M1-4 定义内外两层接口合同

**目标：**
先把系统边界定清楚，再进入实现。

**至少应定义：**

* 外部接口：`create_task`、`run_coder`、`run_reviewer`、`advance_state`、`approve_human_review`
* 内部接口：`exec`、`read_final_output`、`parse_events`

**完成标准：**

* 每个接口的输入输出结构已明确
* 错误返回形态已明确
* 业务层与 CLI 细节之间已有清晰隔离

### 5.3 M2：Mock 与测试基线

M2 的目标是先把“系统如何被证明正确”定下来，再进入实现。

#### M2-1 设计 mock 剧本

**目标：**
把首版所有关键裁决场景先收敛成可重复的 fixture。

**最低剧本：**

* coder 成功，reviewer clean
* reviewer 给出 in-scope blocking finding
* checks 未过导致回环
* reviewer 输出 `needs_human_ruling`
* loop 超限触发人工升级
* 人工批准后结案

**完成标准：**

* 每个剧本都有对应输入与期望输出
* 剧本可直接用于状态机测试
* 剧本不依赖真实 Codex 运行

#### M2-2 编写 schema 校验测试

**目标：**
让四个核心模型先被测试锁住。

**完成标准：**

* 合法对象能通过校验
* 缺关键字段时会失败
* 关键枚举值与边界条件有覆盖

#### M2-3 编写状态机测试

**目标：**
先验证 `advance_state()` 的业务裁决，再写实现。

**完成标准：**

* clean / needs_fix / needs_human_ruling 都有断言
* loop 超限有断言
* depends_on 或 must-do 不满足时有明确行为断言

#### M2-4 编写接口合同测试

**目标：**
锁住 adapter 的输入输出边界，防止实现阶段接口漂移。

**完成标准：**

* adapter 返回结构满足约定
* 失败路径有统一错误对象
* 结果文件读取与解析行为可测试

### 5.4 M3：实现单轮闭环与自动回环

M3 的目标是在 M2 测试约束下完成实际功能。

#### M3-1 实现 Codex CLI adapter

**目标：**
统一封装 `exec`、输出路径、schema 注入、模型与权限参数。

**完成标准：**

* 业务代码不直接拼 `codex exec`
* 关键参数可显式传入
* stdout / stderr / exit code / 结果文件可被记录

#### M3-2 跑通 coder 单轮执行

**目标：**
从 `task_packet` 装配 coder 上下文，拿回一个满足 schema 的 `coder_result`。

**完成标准：**

* coder 结果可解析
* 未执行检查可显式标注
* 代码变更与未做项有结构化记录

#### M3-3 跑通 reviewer 单轮执行

**目标：**
在有 `task_packet + coder_result + diff` 的前提下，拿回一个可裁决的 `review_result`。

**完成标准：**

* findings 结构完整
* promotion_readiness 可计算
* 无法判断时可输出 `needs_human_ruling`

#### M3-4 实现 advance_state

**目标：**
把 controller 的核心裁决写成 deterministic 规则。

**完成标准：**

* `needs_fix` 路径可回 coder
* `review_clean` 路径可进入人工审查
* `needs_human_ruling` 可稳定升级人工

#### M3-5 实现 loop 上限与重复 finding 保护

**目标：**
防止自动化退化成无限循环。

**完成标准：**

* 有全局默认最大轮次
* 能检测重复 blocking finding
* 触发后进入人工裁决而不是继续重跑

#### M3-6 实现人工签字入口

**目标：**
保留系统法理上的最终裁决点。

**完成标准：**

* 存在明确的 `approve_human_review(task_id)` 入口
* 只有人工批准后才能进入 `DONE`
* 结案摘要可回写到 `task_state`

### 5.5 M4：真实集成与试点收口

M4 解决的是“mock 下成立”和“真实运行下成立”之间的差距。

#### M4-1 真实 Codex 集成 smoke test

**目标：**
用最小真实运行验证 adapter 和结果读取链路确实可用。

**完成标准：**

* 至少有 1 条真实 `codex exec` 路径可跑通
* 最终结果可按 schema 落盘并被读取
* 真实运行失败时能产出可诊断错误信息

#### M4-2 补齐少量端到端验证

**目标：**
确认 mock 之外，系统在真实环境下也能完成最小闭环。

**最低场景：**

* 至少一条 `NEW -> DONE`
* 至少一条 `NEW -> NEEDS_HUMAN_RULING`
* 至少一条 `NEW -> NEEDS_FIX -> REVIEW_CLEAN`

#### M4-3 选一个真实 milestone 试点

**目标：**
在真实但低风险的连续任务链中验证首版有效性。

**试点要求：**

* 任务边界清晰
* checks 稳定可执行
* 有明确责任人愿意人工签字
* 允许在失败时回到人工流程

## 6. 首版推荐任务链

为了方便后续直接生成 `task_packet`，建议把首版建设任务按如下顺序推进。

| 顺序 | 任务 ID | 任务名称 | 依赖 |
| --- | ----- | ---- | -- |
| 1 | `P0_repo_scaffold` | 仓库脚手架与目录骨架 | 无 |
| 2 | `P1_schema_baseline` | 四个核心 schema 基线 | `P0_repo_scaffold` |
| 3 | `P2_state_machine` | 状态枚举、跃迁表与 task_state 持久化 | `P1_schema_baseline` |
| 4 | `P3_interfaces_contract` | 内外两层接口合同 | `P2_state_machine` |
| 5 | `P4_mock_scenarios` | mock 剧本与 fixture | `P3_interfaces_contract` |
| 6 | `P5_contract_tests` | schema、状态机、接口合同测试 | `P4_mock_scenarios` |
| 7 | `P6_codex_adapter` | Codex CLI adapter 与运行落盘 | `P5_contract_tests` |
| 8 | `P7_coder_reviewer_flow` | coder/reviewer 单轮执行链路 | `P6_codex_adapter` |
| 9 | `P8_controller_loop` | advance_state、回环、升级人工 | `P7_coder_reviewer_flow` |
| 10 | `P9_human_approval_and_pilot` | 人工签字、smoke test 与试点联调 | `P8_controller_loop` |

### 6.1 为什么按这个顺序

这个顺序背后的原则有四条：

* 先立合同，再接运行时
* 先写 mock 和测试，再写实现
* 先跑通单轮，再做自动回环
* 先证明可裁决，再追求可观测与可扩展

一旦颠倒顺序，就会很容易出现两类问题：

* adapter 已经写了很多，但 schema 还没定，结果调用层频繁返工
* prompt 与流程已经跑起来，但 controller 没有稳定裁决依据，最后还是回到人工搬运
* mock 和测试还没收敛，真实集成一变动，系统就无法判断是模型问题还是合同问题

## 7. 每类任务的 task packet 模板建议

为了让后续任务拆分真正能被系统消费，建议每个建设任务都按统一模板落地。

### 7.1 必填字段

每个任务至少要回答以下问题：

* `title`：当前任务是什么
* `entry_docs`：依据哪些文档
* `must_do`：必须交付什么
* `must_not_do`：明确不做什么
* `done_criteria`：什么条件下算完成
* `required_checks`：必须执行哪些验证
* `depends_on`：依赖哪些前置任务

### 7.2 对首版特别重要的两个字段

#### 一、must-not-do

这字段在首版尤其重要，因为系统建设阶段最容易 scope 膨胀。

例如在 `P3_codex_adapter` 中，`must_not_do` 就应该明确写：

* 不接 CI
* 不实现多模型动态路由
* 不支持多任务队列

#### 二、done criteria

done criteria 必须可执行，而不是口头完成。

坏例子：

* adapter 基本可用
* reviewer 看起来差不多

好例子：

* 可通过统一接口触发 `codex exec`
* 最终结果可按 schema 落盘并读取
* 失败时返回结构化错误对象

---

## 8. 关键验收门槛

任务规划如果没有阶段门槛，最终只会变成“做了很多事”而不是“系统已经成立”。

### 8.1 M1 验收门槛：合同成立

达到以下条件才算通过 M1：

* 四个 schema 已稳定存在
* 状态枚举与跃迁规则可测试
* `task_state` 能承载最小运行态
* 两层接口合同已明确

### 8.2 M2 验收门槛：测试基线成立

达到以下条件才算通过 M2：

* mock 剧本已覆盖核心裁决路径
* schema、状态机、接口合同测试已存在
* controller 的关键规则已先被测试锁住

### 8.3 M3 验收门槛：闭环成立

达到以下条件才算通过 M3：

* coder 能产出合法 `coder_result`
* reviewer 能产出合法 `review_result`
* `needs_fix` 能自动回 coder
* `clean` 能进入人工审查
* `needs_human_ruling` 能停止自动流程并升级人工
* loop 超限不会继续空转

### 8.4 M4 验收门槛：试点成立

达到以下条件才算通过 M4：

* 至少一个真实任务链完成试跑
* 中间状态无需人工搬运维护
* 责任人能基于结构化产物完成最终签字

---

## 9. 资源安排与角色建议

首版不需要大团队，但需要角色清晰。

| 角色 | 主要责任 |
| --- | ---- |
| **产品 / 方案责任人** | 决定任务边界、批准规划、确认试点范围 |
| **技术负责人** | 封板 schema、状态机、目录布局与默认安全策略 |
| **平台开发者** | 实现 controller、adapter、状态持久化与测试 |
| **人工审查责任人** | 承担 `HUMAN_REVIEW` 的最终裁决 |

### 9.1 不建议的协作方式

以下协作方式会直接削弱首版推进效率：

* 没有明确技术负责人，所有字段大家边做边改
* 由 reviewer prompt 反向决定 controller 逻辑
* 在首版尚未跑通前就并行推进 dashboard、CI、通知中心

## 10. 当前主要风险与对应规划动作

### 10.1 schema 漂移

**风险：**
文档、prompt、controller 规则各写一套，最终字段名或语义不一致。

**规划动作：**

* 先完成 `P1_schema_baseline`
* 之后所有实现都以 schema 为先
* prompt 只能消费 schema，不得自创业务字段

### 10.2 controller 过胖

**风险：**
controller 在实现中开始读取大量自由文本并重做审查判断。

**规划动作：**

* 在 `P6_controller_loop` 里只实现状态裁决
* 任何代码质量判定都必须来自 `review_result`

### 10.3 过早平台化

**风险：**
还没证明单任务闭环成立，就开始做多任务、可视化、服务化。

**规划动作：**

* 把这些能力明确列入 v1 非目标
* 只有 `P9_pilot_run` 通过后才讨论扩展项

### 10.4 试点任务选错

**风险：**
选择边界模糊、依赖复杂、验证不稳定的任务做首个试点，导致对系统产生错误负反馈。

**规划动作：**

* 试点任务必须边界清晰
* 必须有稳定 checks
* 必须允许失败后退回人工流程

---

## 11. 推荐落地顺序

如果从下周开始推进，建议按如下顺序执行，而不是并行铺开：

1. 先完成规划封板与目录骨架。
2. 再定义四个 schema、状态跃迁和两层接口合同。
3. 然后先写 mock 剧本与核心测试。
4. 接着实现 Codex CLI adapter。
5. 再打通 coder、reviewer 两条单轮链路。
6. 然后写 controller 的自动回环与人工升级。
7. 最后补真实 smoke test 与试点联调。

这个顺序的意义在于：

> **先让系统具备法理，再让系统具备动作；先让系统能正确停止，再让系统追求跑得更远。**

---

## 12. 结论

对当前项目而言，任务规划文档真正要做的，不是把系统说得更宏大，而是把首版压缩成一条可执行路径：

> **以单仓库、单任务串行、四个核心模型、两层接口合同、合同优先 TDD 为边界，先跑通一个真实 milestone 的自动闭环。**

因此，首版规划的判断标准不应是“功能是否足够多”，而应是三件事：

* 任务边界是否稳定
* 状态推进是否可执法
* 测试与真实试点是否都可跑通

只要这三件事成立，后续再扩展并行调度、可视化、CI 接入，系统才有健康演进的基础。

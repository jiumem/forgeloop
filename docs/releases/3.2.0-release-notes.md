# Forgeloop 3.2.0

Forgeloop 3.2.0 让采用 `SHARED` Branch Topology 的 Spec 直接由 Spec Root 承担最终集成门禁，移除仅用于复验、创建 PR/MR 或合并的特殊最终 Ticket。

## 主要改进

- 为四种合法共享分支原因统一声明 `Final integration gate owner: SPEC_ROOT`；
- 在竞争 root Claim 前为每个共享 Spec 冻结 `spec_delivery_base`，并由不可变 `RUN_CLAIMED` Payload 精确绑定；
- 在全部普通 Ticket 关闭并写入 Integration Result 后，由唯一 Final Integration Gate 冻结和核验完整交付范围；
- 按原生 integration method 验证 merge、squash 和其他已配置方法的 Candidate 到交付 Commit 映射；
- 通过 `ALREADY_PRESENT` Integration Result 映射合法零变更 Ticket，不创建空 Commit；
- 最终验证复用父级 Validation Entries 与既有 Cross-seam Invariant Owner Proof，不新增 Reviewer、Verdict 或事实源；
- 最终阶段只有在存在独立实现 Scope 时才创建普通纵向 Ticket；
- Final Gate Finding 通过既有 `RUN_PAUSED` 和显式 `to-tickets` Repair Mode 路由，保持已完成 Ticket 与 Checkpoint 不可变；
- `human-merge` 保持 Spec Open且没有虚构的当前 Ticket；`auto-merge` 只在固定交付 Head、当前目标与原生检查全部成立后执行；
- Spec Integration Result 与 fresh Spec Acceptance 继续分别承担集成事实和最终验收。

## 不兼容变更

- 3.2.0 不兼容旧 `Final integration owner` 声明；
- 缺失 `Final integration gate owner: SPEC_ROOT` 或仍包含 ceremony-only 最终 Ticket 的共享图返回 `FAILED_PRECONDITION`，Tracker 零写入；
- 不提供旧字段 alias、parser、自动迁移、兼容分支或运行时回退；
- 尚未完成的旧式 ceremony Ticket 只能在用户明确批准 Spec Revision Reconciliation 后标记为 superseded并清理原生关系。

## 架构边界

- Final Integration Gate 是只读 Agent 门禁，不是 Ticket、Event、状态机、Reviewer 类型或 Acceptance level；
- Tracker、Git、PR/MR 与目标分支继续保存原生事实，提示词不维护平行程序状态；
- Spec Root、Scheduler 和 Gate 验证任务不能修改实现代码或解决代码冲突；
- 20 个正式 Skill、11 个用户入口与 9 个模型可调用能力保持不变。

## 验证

- Release Validator、Runtime Contract、元数据、上游同步和 Plugin Validator；
- GitHub、GitLab、Local 的共享交付 Fixture；
- Python 全量单元测试与显式 Agent 评测；
- 本地 Marketplace 候选重装、Installed Validator、源码与缓存逐文件比对；
- Release Tag、GitHub Release 与最终 `main` Commit 精确回读。

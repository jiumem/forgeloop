# Forgeloop 3.4.0

Forgeloop 3.4.0 将三轮 Repair Budget 从永久停止点改为强制语义诊断边界，并让需要改变正式契约的执行阻塞通过一次用户批准完成调和与恢复。

## 主要改进

- 每个修复周期仍最多执行三轮 Candidate-changing 修复；第三轮后仍失败时，必须先发布并精确回读 `RUN_PAUSED / REPAIR_BUDGET`；
- 由 fresh Coder 基于完整契约、累计 Candidate、双轴 Findings、历次诊断和验证证据执行语义化 Exhaustion Diagnosis；
- 可信且实质不同的 Scope 内新机制可自动开启下一组三轮，继续使用同一 Ticket、Run 和 Branch，无需用户批准或 replacement Ticket；
- 新周期使用 fresh Coder 与 fresh 双 Reviewer，同一周期内继续复用对应 Agent 上下文；
- 没有可信新机制或真实净进展时进入 `IMPLEMENTATION_BLOCKED`，无条件“继续试试”不能恢复执行；
- 需要改变 Spec、Scope、Acceptance、ADR 或批准接口时进入零预算 `CONTRACT_BLOCKER`；
- 协调 Agent 在询问用户前形成完整契约调和包并完成内部语义 Design Review；
- 用户一次选择即可拒绝、批准后暂停，或批准并在完整原生回读后竞争恢复原 Run；
- `to-spec`、domain modeling、`to-tickets` 和 `run-initiative` 保持各自事实所有权，不建立第二套契约数据库或程序状态机；
- 调和优先保留和更新 Open Tickets，已确认写入只向前恢复，Completed/Closed 历史和旧 Candidate Git 证据保持不可变；
- Candidate 与双 Verdict 绑定有效 Spec/Ticket/ADR Revision；影响当前 Ticket 的 material Revision 可开启新周期，无关或非实质修改不能重置预算；
- 自动修复不设置任意总周期上限，以证据支撑的新机制和真实进展作为持续条件。

## Agent 原生边界

- 诊断与调和可以使用稳定语义字段组织推理、恢复和审计；
- 字段内容由 Agent 结合完整上下文理解，Scheduler 不使用 parser、正则、关键词、评分或布尔值代替语义判断；
- `package_id`、Revision、Run、Ticket 和原生顺序键只承担授权、幂等、绑定与冲突检测；
- 不新增 replacement Ticket 类型、Event、Reviewer 类型、终态、工作流 DSL 或第二事实来源。

## 兼容性

- 3.4.0 保持 20 个正式 Skill、11 个用户入口和 9 个模型可调用能力不变；
- 不需要从 3.3.1 执行名称迁移；
- 自定义 `run-initiative` Prompt 或复制的运行协议应同步新的修复周期、`IMPLEMENTATION_BLOCKED` 和一次契约调和语义。

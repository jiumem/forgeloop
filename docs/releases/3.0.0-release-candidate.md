# Forgeloop 3.0.0 发布候选

状态：发布候选门禁已通过并取得用户发布授权；仅允许从合并到 `main` 的最终 Commit 发布。

## Breaking Change 摘要

- 活动套件由五个本地文档型 Skills 原子替换为 20 个 Tracker 驱动 Skills。
- GitHub、GitLab 与 Local Markdown 成为 Spec、Ticket、Blocking 与运行状态的唯一真理源。
- `run-initiative` 只接受正式 Spec 或持久化 Initiative 引用，使用独立 Coder、Spec Reviewer、Standards Reviewer、修复预算、集成与分层 Acceptance。
- 旧名称、历史产物处理、两条活动迁移路径和回退见 [迁移指南](../migrations/2.5.0-to-3.0.0.md)。

## 验证证据

- 2.5.0 基线 Commit：`a32f7a5d1eceace7215321dd7168dc7d07dac249`，可通过 Git 归档恢复五 Skill 源码。
- 活动目录：恰好 20 Skills；11 个仅显式调用，9 个模型可调用。
- Fixture：9 项入口/Tracker Fixture + 32 项 Runtime Fixture + 4 项 Recommend Initiatives Fixture。
- 结构：20/20 Skill Creator、套件发布态、Plugin Creator、上游漂移、UI 元数据、运行协议、57 项 Python 单元测试和文档链接检查通过。
- 安装：本地 `forgeloop-local` Marketplace 以 `3.0.0+codex.20260712111912` 重装并通过 Installed Validator；发布源码已经恢复严格版本 `3.0.0`。
- 新任务：宿主实际暴露恰好 9 个模型可调用 Skill；`setup-forgeloop`、`to-spec`、`to-tickets`、`run-initiative` 均未隐式注入。显式调用缺少正式 Spec 的 `to-tickets` 返回 `FAILED_PRECONDITION`，未自动启动 Setup，也未回退 Local。

## 最终发布命令

仅在 Release PR 合并后执行；先把 `<release-commit-sha>` 替换为最终 `main` Commit。

```bash
gh release create v3.0.0 \
  --repo jiumem/forgeloop \
  --target <release-commit-sha> \
  --title "Forgeloop 3.0.0" \
  --notes-file docs/releases/3.0.0-release-notes.md \
  --latest
```

Tag 必须指向合并后的最终 `main` Commit；不得从本地候选提交或未合并分支发布。

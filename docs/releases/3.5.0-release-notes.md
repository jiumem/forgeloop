# Forgeloop 3.5.0

Forgeloop 3.5.0 把方案必要性审计放到正式 Spec 发布之前：Agent 必须先证明复杂度服务于当前失败或正式契约缺口，才能把它写入下游必须执行的契约。

## 主要改进

- `to-spec` 在首次 Tracker 写入前，对完整候选 Spec、已解决上下文、代码事实和相关 ADR 执行语义化 Necessity Review；
- Review 以一段连续的 Agent 推理程序工作：从当前阻塞和失败机制出发，定位最小权威生产 Seam，盘点并复用既有事实、owner、接口和生命周期；
- 每个新 token、lease、version、registry、事实源、生命周期或状态模型都必须绑定当前已复现失败，或由真实运行条件证明的权限、并发、恢复、兼容或不可逆失败模式；
- 更小方案只有在无法满足当前 Delivery Acceptance、Cross-seam Invariant 或必要失败行为时才能被拒绝；未来用途、无当前消费者的 helper 和未来专用基础设施不能作为证据；
- 旧机制退出 Proof 以受影响生产路径为边界，只有正式方案明确全局淘汰旧机制时才要求合法生产调用为零；
- 既有授权支持更小方案时，Agent 会删除无证据复杂度并重新形成完整候选；修改后从用户批准的测试 Seam 起重跑所有发布门，不复用旧的 Acceptance、Validation、Prerequisite 或 Invariant 判断；
- 更小方案需要新的产品选择、Scope、失败语义、ADR 或批准接口决定时，复用 `CONTEXT_INSUFFICIENT`，不替用户发明决定；
- 必要性证据继续自然落入现有规范内容，不新增 Necessity schema、强制章节、parser、固定评分、Reviewer、Verdict 或状态机。

## Ticket 最小性

- `to-tickets` 只忠实拆分已批准 Spec，不重复完整 Necessity Review；
- 一个 Spec 默认拆成一至三个最小完整纵向 Ticket；这是可由证据推翻的强默认，不是程序上限；
- 每个 Ticket 合并后必须形成当前可观察的用户或系统结果；未来专用基础设施、无当前消费者的 helper 和只有“以后可能需要”理由的 Ticket 不得发布；
- 必要 enabling change 与其当前消费者合并；prefactoring 保留在首个消费它的纵向 Ticket 内。只有它们自身提供当前必需且可观察的系统保证时才可独立成票；
- 超过三个 Ticket 时，Agent 必须对每个额外 Ticket 说明当前关闭的结果，以及省略后无法成立的 Acceptance、Invariant 或必要失败行为；
- 正确拆票若需要改变正式 Problem、Actor、Scope、Delivery Acceptance、ADR、产品行为或公开接口，返回 `CONTRACT_BLOCKER`。

## 兼容性

- 3.5.0 保持 20 个正式 Skill、11 个用户入口和 9 个模型可调用能力不变；
- `spec-standards-review` 的调用描述明确支持只有仓库标准、没有 originating Spec 的合法审阅；探索、影响分析和调试仍由各自能力处理；
- 不新增 Tracker 状态、运行事件、配置迁移或名称迁移；
- 已发布 Spec 不会被自动迁移或改写；新规则用于后续候选 Spec 和 Ticket 图。

## 验证

- Necessity Review 提示词契约测试与批量 authenticated Codex Agent evaluation；
- 简单方案收敛、有证据结构性例外、缺少用户决定、受影响路径退出、全局淘汰和拆票契约漂移场景；
- Release Validator、Runtime Contract、Skill UI 元数据、上游同步和 Plugin Validator；
- Python 全量单元测试与本地 Marketplace 候选重装；
- Release Tag、GitHub Release 与最终 `main` Commit 精确绑定。

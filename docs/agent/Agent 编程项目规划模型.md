# Agent 编程项目规划模型

## 方案设计、差距分析与总任务文档的正式规划法

> **状态**：Sealed / v1
> **定位**：Agent 编程项目的正式规划法
> **适用范围**：大任务、强规划、快开发、分层验证与审查的软件工程项目
> **不适用范围**：一次性脚本、短期实验、无需主干收敛与发布纪律的临时代码
> **核心目标**：把真实需求在进入实现前，压缩为一份**单篇总任务文档**；该文档应同时具备**执行对象、动态编排、集成路径、验收矩阵与风险账本**，作为后续 Planner / Coder / Reviewer 与 Codex 执行循环的合法输入

---

## 0. 文档定位

本文档只回答一个问题：

> **如何把一个需求，规划成一份正式的总任务文档。**

这里的“总任务文档”不是待办清单，不是产品 PRD，也不是设计说明的简写版。它是一个已经承接了前置裁决结果的**执行总图**。没有这张总图，后续实现再快，也只会把局部正确堆成系统性熵增。

本文档与上位文档的关系必须先钉死：

| 文档         | 法位   | 本文档与其关系                                                            |
| ---------- | ---- | ------------------------------------------------------------------ |
| **执行模型法典** | 上位宪法 | 本文档只继承，不重定义 Initiative / Milestone / Task / PR / Gate / Review 的法位 |
| **核心术语表**  | 上位词典 | 本文档只使用，不重新发明术语                                                     |
| **本规划模型**  | 规划法  | 定义“如何从前置设计进入总任务文档”                                                 |

因此，本文档**不讨论**以下内容：

* Prompt / Skill 文本如何写
* 具体 Codex 命令如何操作
* 具体 CI 平台与脚本名
* 具体仓库目录布局
* 具体实现细节与代码级设计展开

这些内容应在角色协议文档、Codex 落地文档或仓库级规则中继续展开，而不应污染规划法本体。

---

## 1. 核心结论

整套规划模型可以压缩为一句话：

> **规划不是列待办，而是形成一份单篇总任务文档。**

这份文档必须把一个已完成前置裁决的需求，压缩成如下执行结构：

> **Initiative → Milestone 总表 → Workstream → Task Ledger → 分支与 PR 集成路径 → 验收矩阵 → 全局残余风险与后续动作**

这里最关键的不是信息量，而是法位顺序。整套方法论的基础是：

| 层次          | 本质                                       |
| ----------- | ---------------------------------------- |
| **静态执行对象**  | Initiative / Milestone / Task            |
| **动态编排结构**  | Workstream / 依赖关系 / Agent 分配 / 分支与 PR 路径 |
| **成立与裁决依据** | 验收矩阵 / 残余风险 / 前置文档引用                     |

因此，规划阶段真正产出的不是“任务描述”，而是**执行图**。
这张执行图必须能够同时支持四件事：

* 让 Planner 证明边界已经收口
* 让 Coder 按图索骥，自主查阅前置文档并进入实现
* 让 Reviewer 在有法定上位参考的前提下做裁决
* 让主干集成、阶段验证与最终交付拥有统一坐标系

## 2. 三类规划文档及其关系

需求不是直接进入总任务文档。
在 Agent 编程里，真正成熟的顺序必须是：

> **先方案设计。**
> **重构 / 迁移类需求再做差距分析。**
> **最后才生成总任务文档。**

这三类文档的关系，必须被正式写死。

---

### 2.1 方案设计文档

方案设计文档是所有大任务的前置文档。
它的职责不是拆任务，而是裁决方向。

它至少回答以下问题：

| 问题       | 含义                |
| -------- | ----------------- |
| **为什么做** | 当前问题与目标动机是什么      |
| **做什么**  | 目标结构、能力边界、约束范围是什么 |
| **不做什么** | Non-Goals 是什么     |
| **怎么定**  | 关键方案、结构裁决、边界原则是什么 |

因此，方案设计文档的产出是：

> **目标态与方案裁决。**

它不是总任务文档的替代品，也不是后续 PR 列表。

---

### 2.2 差距分析文档

差距分析文档只在**重构、迁移、替换、治理收敛**类需求中需要。
它的职责不是再讲愿景，而是建立现状与目标态之间的执行桥梁。

它至少回答以下问题：

| 问题          | 含义                    |
| ----------- | --------------------- |
| **现状是什么**   | 当前系统真实状态、既有路径、现存约束    |
| **目标态是什么**  | 方案设计文档定义的目标结构         |
| **差距在哪里**   | 数据、接口、状态、责任层、兼容路径上的缺口 |
| **迁移闭环是什么** | 哪些差距先补、哪些后切、哪些必须并行治理  |

因此，差距分析文档的产出是：

> **从现状到目标态的差距账本与迁移视图。**

它不是任务文档，更不是 review 评论的集合。

---

### 2.3 总任务文档

总任务文档是执行入口文档。
它不再负责方案思考，也不再负责差距识别，而是承接已经封板的设计与差距裁决，展开完整执行图。

它回答的是：

> **这个 Initiative 将被切成哪些 Milestone。**
> **每个 Milestone 下沿哪些 Workstream 推进。**
> **每条线上的 Task 是什么。**
> **这些 Task 如何进入分支与 PR 集成路径。**
> **各层到什么程度算成立。**

进入实施后，这份总任务文档默认绑定 **一个 Initiative** 作为正式执行入口。
后续执行体系应围绕 Initiative 选择当前 Milestone frontier 与 Ready Tasks；Task 作为内部原子被调度，而不是终端用户的常态主入口。

因此，三类文档之间的法位关系是：

| 文档         | 法位   | 进入下一文档的前提             |
| ---------- | ---- | --------------------- |
| **方案设计文档** | 方案裁决 | 关键结构与边界已定             |
| **差距分析文档** | 迁移裁决 | 仅在重构/迁移类需求中，现状与目标差距已定 |
| **总任务文档**  | 执行总图 | 前置裁决已完成，不再携带未决问题      |

---

## 3. 总任务文档的写作原则

这篇规划模型最容易写歪的地方，不在字段设计，而在写作姿态。
总任务文档一旦写成“第二份设计文档”或“给 Agent 的操作说明书”，整个体系就会立即长出双真值。

因此，总任务文档必须遵守以下原则。

---

### 3.1 薄文档原则

> **总任务文档是执行索引文档，不是实现说明文档。**

它的职责不是长篇复述方案背景，也不是教 coder 应该怎样写代码。
它只负责三件事：

| 职责     | 含义                                            |
| ------ | --------------------------------------------- |
| **索引** | 指向方案设计文档、差距分析文档、规范切片                          |
| **收敛** | 定义 Initiative / Milestone / WS / Task / PR 路径 |
| **裁决** | 定义边界、依赖、验收与残余风险                               |

因此，Coder 的职责不是等任务文档把一切讲完，而是：

> **按图索骥，自行查阅前置文档，理解背景与边界，然后在既有法位内完成实现。**

---

### 3.2 固定字段原则

> **总任务文档按固定字段写，不按自由散文写。**

这意味着它追求的不是“写得越多越好”，而是：

* 字段齐全
* 语义稳定
* 长度受控
* 可扫描
* 可索引
* 可比较

“全部细化”的正确含义也必须在这里钉死：

> **全部细化指所有对象、依赖、验收、路径都必须显式存在并拥有完整字段。**
> **它不等于每个 Task 都写成长篇实现教程。**

因此，总任务文档应当像控制面板，而不是论文正文。

---

### 3.3 静态对象先于动态编排

静态执行对象是骨架，动态编排结构是挂在骨架上的作战图。
顺序绝不能反。

> **先有 Initiative / Milestone / Task，后有 Workstream / 依赖图 / Agent 分配 / PR 路径。**

一旦先谈并行、先谈 PR、先谈分工，而对象边界尚未成立，后续编排就只是在放大语义模糊。

---

### 3.4 未裁决问题禁入原则

> **未裁决问题可以留在方案设计文档或差距分析文档中，不得进入总任务文档，更不得进入实现。**

总任务文档的前提是：
关键方案、关键边界、关键迁移问题都已经裁决完成。

因此，总任务文档中不存在“Open Decisions”区块。
如果还有未决问题，正确动作不是开始拆 Task，而是返回前置文档继续裁决。

---

### 3.5 多 Milestone 优先原则

> **默认优先多切 Milestone，而不是在同一 Milestone 下堆多个 PR。**

原因很简单：

* Milestone 是状态收敛边界
* PR 是物理集成路径

如果一个里程碑需要靠多个 PR 才勉强可合入，首先应怀疑的是 Milestone 是否切得过厚，而不是先默认进入多 PR 模式。

因此，本规划法的默认偏好是：

> **先切状态断点，再切集成路径。**

多 PR 只作为例外机制存在，而不是默认形态。

---

本章给出**单篇总任务文档**的正式模板。
它既是字段标准，也是后续实施—验证—审查循环的合法输入协议。

### 4.2 前置输入与决策基线

这一节不是背景复述，而是引用与裁决摘要。
它的职责是证明：总任务文档不是拍脑袋生成，而是继承已封板前置文档。

| 字段                      | 含义               |
| ----------------------- | ---------------- |
| **Requirement Summary** | 当前需求的最小摘要        |
| **Design Refs**         | 方案设计文档引用         |
| **Gap Analysis Refs**   | 差距分析文档引用，仅在适用时出现 |
| **Sealed Decisions**    | 已封板关键裁决摘要        |
| **Execution Boundary**  | 当前执行边界的总裁决       |
| **Initiative Reference Assignment** | 当前专项进入 G3 / R3 时的法定参考入口指派 |

建议模板：

```markdown

- 目标摘要：

### 1.2 Design Refs
- <方案设计文档链接 / 索引>

### 1.3 Gap Analysis Refs
- <差距分析文档链接 / 索引>
- 若不适用，写明：N/A

### 1.4 Sealed Decisions
- 决策 1：
- 决策 2：
- 决策 3：

### 1.5 Execution Boundary
- 本总任务文档承接的执行边界：
- 明确不再在本文档中重开裁决的问题：

### 1.6 Initiative Reference Assignment
- 当前 Initiative 的法定参考入口：
- 允许承载形式：方案设计文档 + 总任务文档，或显式指定的 Initiative 参考文档
- 若存在多份相关文档，写明唯一指派入口：
```

> **默认 Single PR。**

这部分是动态编排核心。
它不创造新对象层，只负责说明当前 Initiative 如何组织并行作战。

```markdown
|---|---|---|---|
| WS-A |  |  |  |
| WS-B |  |  |  |
| WS-C |  |  |  |

- WS-A 边界：

### 4.3 Dependency Graph
- WS 依赖关系：
- 关键 Task 依赖关系：
- 关键里程碑汇合点：

### 4.4 Agent Allocation
| Line / WS / PR | Recommended Executor | Parallelizable | Depends On |
|---|---|---|---|
| WS-A | Agent-A / User / Shared |  |  |
```

* Agent-A / Agent-B / Agent-C
* User
* Shared
* Review-first

这样规划输出才真正从“结构图”进入“作战图”。

```markdown
- Dependencies:
- Acceptance:
- Task-local Risks / Notes:

#### T002
- Design Refs:

- Spec Refs:
```

第二，`Action` 只写要改什么，不写实现教程。

> **总任务文档负责定义做什么与边界在哪。**
> **Coder 负责依据前置文档决定如何实现。**

### 4.7 分支与 PR 集成路径

这一节承接上位宪法定义，给出逻辑集成路径与推荐命名法。

```markdown
|---|---|---|---|---|---|
| PR-01 | M1 |  |  |  |  |
| PR-02 | M2 |  |  |  |  |

### 6.4 PR Dependency Order
- 串行 PR：

- 汇合 PR：
```

推荐默认命名法如下：

| 对象                     | 推荐命名                            |
| ---------------------- | ------------------------------- |
| **Milestone 分支**       | `m1-<slug>`                     |
| **带 Initiative 前缀的分支** | `<initiative-prefix>/m1-<slug>` |
| **Milestone PR 标题**    | `[M1] <summary>`                |

| **多 PR 例外模式标题**        | `[M1][PR-01] <summary>`         |

但必须再次强调：

> **命名规范从属于状态收敛，不得反向替代对象切分。**

---

### 4.8 验收矩阵

验收不是文末空泛总结，而是整个执行图的成立索引。
Task 验收在 Task 字段里已存在，Milestone 验收在总表里已存在，Initiative 成功标准也在 Initiative 区块里已存在。这里再给一张总览矩阵，其作用是**总索引**，而不是重新叙述。

```markdown
## 7. Acceptance Matrix

| Level | Object | Acceptance | Evidence Entrypoint |
|---|---|---|---|
| Task | T001 |  | G1 / R1 / tests / spec refs |
| Task | T002 |  | G1 / R1 / tests / spec refs |
| Milestone | M1 |  | G2 / R2 / PR / integration evidence |
| Milestone | M2 |  | G2 / R2 / PR / integration evidence |
| Initiative | INIT |  | G3 / R3 / release / rollout / deployment |
```

这里还必须钉死一条工程铁律：

> **任何对象的成立，都不能只靠 Agent 生成的本地证据自证。**
> **尤其是 Milestone 与 Initiative 的成立，必须接受更高层的验证与真实环境裁决。**

---

### 4.9 全局残余风险与后续动作

这里不再允许出现未裁决问题。
它只记录两类东西：

* 已知且当前接受的全局残余风险
* 明确留给后续里程碑、后续专项、后续治理项的动作

```markdown
## 8. Global Residual Risks & Follow-Ups

### 8.1 Global Residual Risks
- 风险 1：
- 风险 2：
- 风险 3：

### 8.2 Follow-Ups
- Follow-Up 1：
- Follow-Up 2：
- Follow-Up 3：
```

这里的风险必须满足三个条件：

| 条件      | 含义                                   |
| ------- | ------------------------------------ |
| **已知**  | 不是尚未思考的问题                            |
| **可解释** | 能说明影响范围与接受原因                         |
| **可追踪** | 能被后续 Milestone / Initiative / 治理动作承接 |

---

## 5. 填写规则

本章只回答一个问题：

> **每一栏怎么填，才不会把总任务文档写成废纸。**

---

### 5.1 前置文档填写规则

前置文档区块必须做到三件事：

第一，显式索引。
第二，显式继承。
第三，显式指派法定参考入口。

错误写法是“背景详见前文”。
正确写法应明确列出：

* 方案设计文档索引
* 差距分析文档索引
* 关键决策摘要
* 当前执行边界
* Initiative Reference 的唯一指派入口

这里的摘要应压缩为**决策基线**，而不是再讲一遍前置文档。

---

### 5.2 Initiative 填写规则

Initiative 区块只写专项边界，不写施工细节。
它的重点不是“我们将实现哪些模块”，而是：

* 当前问题是什么
* 本专项覆盖什么
* 本专项明确不覆盖什么
* 如何判定专项成功

若这一节无法写得清楚，后续 Milestone 一定会漂移。

同时，Initiative 区块或前置基线区块必须显式回答：

> **当前专项在进入 G3 / R3 时，究竟以哪一组文档作为法定参考入口。**

---

### 5.3 Milestone 填写规则

Milestone 必须按状态成立来写，而不是按施工顺序来写。
因此，以下写法应直接判为不合格：

* 完成数据库设计
* 完成接口开发
* 完成联调
* 完成页面开发

这些都只是动作，不是状态。

合格的写法应优先表达：

* 哪个能力闭环成立
* 哪个阶段状态首次成立
* 哪个不变量组合可被正式宣布成立

此外，每个 Milestone 还必须显式指派其法定参考入口：

> **要么由总任务文档中的阶段区块承载，要么由独立 Milestone 文档承载；二者可并存，但不得不指派。**

---

### 5.4 Workstream 与 Agent 分配填写规则

Workstream 只能按以下三件事切分：

| 切分标准     | 含义             |
| -------- | -------------- |
| **责任边界** | 谁持有事实源与职责      |
| **耦合关系** | 哪些线可并行，哪些线必须串行 |
| **验证边界** | 每条线如何进入收敛与回归检查 |

因此，错误切法包括：

* 按人头切
* 按页面切
* 按目录切

Agent 分配区块也不能只写“待分配”。
因为并行作战图若没有执行分配，就不是真正的执行图。

---

### 5.5 Task 填写规则

Task 的合法定义必须同时具备以下字段：

| 字段               | 含义                     |
| ---------------- | ---------------------- |
| **Refs**         | 它依赖哪些上位文档与规范切片         |
| **Input**        | 它进入前依赖什么               |
| **Action**       | 它到底改什么                 |
| **Output**       | 它会产出什么                 |
| **Non-Goals**    | 它明确不做什么                |
| **Dependencies** | 它依赖哪些上游 Task / WS / PR |
| **Acceptance**   | 它如何被判定成立               |

这里最关键的一条，是总任务文档对 coder 的边界：

> **Task 不负责教 coder 怎么实现。**
> **Task 只负责定义该实现必须命中的边界、输入、输出与成立条件。**

同时，每个 Task 定义都应足够自洽，能够被后续执行层裁成局部执行包。
不得把关键 Scope、Non-Goals、Acceptance 散落在其他区块，迫使下游执行者靠通读全文去猜当前 Task 边界。

---

### 5.6 分支与 PR 规划规则

PR 规划规则必须继承上位宪法，保持如下顺序：

> **优先切 Milestone。**
> **若里程碑足够清晰且可合入，默认一个 Milestone 一个 PR。**
> **仅在技术性渐进合入时使用多 PR 例外。**

因此，出现多 PR 时，必须先回答一个问题：

> **这是否其实意味着应再切出一个新的 Milestone。**

若答案是肯定的，就不应继续停留在同一里程碑下堆 PR。

---

### 5.7 验收与风险填写规则

验收不是愿景口号。
风险也不是免责声明。

验收必须满足：

* 可判断
* 可举证
* 可映射到 Gate / Review / 真实验证入口

风险必须满足：

* 已知
* 被接受
* 可追踪

总任务文档里绝不允许出现“以后再看”“到时候再定”这类伪风险。
那不是风险，而是未裁决问题；它应被挡在前置文档阶段，而不是流入执行。

---

## 6. 规划流程

模板与规则成立之后，还需要一条稳定流程。
这条流程不是经验顺序，而是法定位次顺序。

---

### 6.1 先做方案设计

所有大任务都必须先做方案设计文档。
没有方案设计，就没有合法的总任务文档起点。

---

### 6.2 重构类需求再做差距分析

若需求属于迁移、替换、治理收敛或大重构，则必须在方案设计之后补做差距分析。
因为这类需求最容易在“现状误判”下直接进入施工。

---

### 6.3 再生成总任务文档

只有当前置裁决已完成，才允许生成单篇总任务文档。
这里的“生成”不是拼接文档，而是将上游裁决压缩为：

* 静态执行对象
* 动态编排结构
* 验收矩阵
* 风险账本

---

### 6.4 最后做实施前封板检查

总任务文档写完后，还不能直接开工。
必须先做一次封板检查，确认它已经具备进入 coder / reviewer / Codex 循环的合法输入质量。

---

## 7. 变更协议

强规划不等于永不变更。
但变更必须有法定入口，不能在实现过程中被悄悄吞掉。

因此，总任务文档封板后，变更必须按级别处理。

| 变更级别                  | 处理方式                          |
| --------------------- | ----------------------------- |
| **Task 内局部调整**        | 可直接修订 Task 字段、PR 路径与验收索引      |
| **Workstream / 依赖变化** | 必须回改总任务文档的并行与集成区块             |
| **Milestone 边界变化**    | 必须回改 Milestone 总表，并重新检查 PR 路径 |
| **Initiative 目标变化**   | 必须返回方案设计文档重新裁决                |
| **现状理解发生变化**          | 对重构 / 迁移类任务，必须回改差距分析文档        |

变更协议的核心目的不是增加流程，而是防止一件事：

> **执行中偷偷漂移，而文档仍假装未变。**

---

## 8. 进入实施的准入条件

一份总任务文档只有同时满足以下条件，才允许进入实施循环。

| 条件                    | 含义              |
| --------------------- | --------------- |
| **前置方案已封板**           | 方案设计文档已完成关键裁决   |
| **差距已封板**             | 适用时，差距分析已完成关键识别 |
| **Initiative 已定义**    | 问题、边界、成功标准已明确   |
| **Milestone 已全量定义**   | 全量阶段断点已明确       |
| **Workstream 已建立**    | 并行作战线与依赖关系已明确   |
| **Agent 分配已给出**       | 并行图已进入作战图       |
| **Task Ledger 已全量细化** | 所有 Task 具备完整字段，且可独立装配局部执行包 |
| **分支与 PR 路径已明确**      | 集成路径不是事后临时拼装    |
| **Reference 入口已指派**     | Milestone / Initiative 的法定参考入口已显式指定 |
| **验收矩阵已存在**           | 所有层级都有成立标准与证据入口 |
| **全局残余风险已登记**         | 已知且接受的风险已显式落账   |

此外，还必须附加一条人工现实约束：

> **真实环境手测与最终合入裁决，仍由用户承担。**
> **Agent 的本地证据只能证明“局部成立”，不能单独证明“交付成立”。**

---

## 9. 禁止事项

为了防止这套方法论在落地时退化，正式列出以下禁止事项。

---

### 9.1 前置阶段禁止事项

* 没有方案设计就直接写总任务文档
* 重构类需求没有差距分析就直接拆 Task
* 未裁决问题流入总任务文档

---

### 9.2 对象层禁止事项

* 把 Milestone 写成施工步骤
* 把 Workstream 升格为第四层主对象
* 把 PR 当作 Initiative 本体
* 把 Branch 当作 Milestone 本体

---

### 9.3 任务层禁止事项

* Task 只有动作，没有 Refs
* Task 只有目标，没有输出
* Task 只有功能，没有验收
* Task 写成实现教程
* 把关键边界散落在 Task 外部，迫使执行者靠猜拼装上下文

---

### 9.4 编排层禁止事项

* 按人头切 Workstream，不按耦合切
* 没有 Agent 分配却宣称可以并行
* 不写依赖关系就宣称存在 PR 序列
* 默认使用多 PR，而不先审视 Milestone 是否应再切分

---

### 9.5 验收层禁止事项

* 验收写成愿景口号
* 风险写成空泛免责声明
* 用“以后再看”替代风险登记
* 让 Agent 本地测试全绿冒充阶段成立或交付成立

---

## 10. 封板结论

本文档可以压缩为以下十二条法则。

### 法则 1

**总任务文档是执行总图，不是待办列表。**

### 法则 2

**需求不会直接生成总任务文档；先做方案设计，重构类需求再做差距分析。**

### 法则 3

**总任务文档是单篇总文档，而不是松散规划包。**

### 法则 4

**总任务文档采用固定字段、强索引、弱叙述的薄文档写法。**

### 法则 5

**静态执行对象先成立：Initiative、Milestone、Task。**

### 法则 6

**动态编排随后挂接：Workstream、依赖关系、Agent 分配、分支与 PR 路径。**

### 法则 7

**默认优先多切 Milestone，而不是在同一 Milestone 下堆多个 PR。**

### 法则 8

**总任务文档必须全量细化，但“全量细化”指字段齐全，不指实现说明无限膨胀。**

### 法则 9

**总任务文档不教 Agent 怎么做；Coder 应依据前置文档自主完成实现。**

### 法则 10

**未裁决问题不得进入总任务文档，更不得进入实现。**

### 法则 11

**任何对象的成立都不能只靠 Agent 本地证据自证；真实环境裁决仍由用户承担。**

### 法则 12

**强规划服务快开发；快开发必须服从分层验证与分层审查。**

---

> **最终收口：**
>
> **Agent 编程的稀缺品，不再是代码。**
> **真正稀缺的是：在高吞吐生成之上，维持边界、依赖、验收与交付的一致收敛。**
>
> 因此，总任务文档不是形式主义。
> 它就是强规划时代的执行控制面。

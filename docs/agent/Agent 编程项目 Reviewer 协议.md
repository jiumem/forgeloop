# Agent 编程项目 Reviewer 协议

> **状态**：Sealed / v1  
> **定位**：Agent 编程项目的正式审查角色协议  
> **适用范围**：需要对差异锚点、阶段收敛与专项交付进行正式裁决的软件工程项目  
> **核心目标**：定义 Reviewer 的**输入协议、输出协议、审查原则、升级规则与交接边界**，使审查活动从“意见表达”收敛为“基于上位参考与证据的正式裁决”

---

## 0. 文档定位

本文档只回答一个问题：

> **Reviewer 在 Agent 编程体系中，究竟审什么，依据什么审，怎么输出，何时升级。**

本文档不提供 Prompt 文案，不提供 `code_review.md` 正文，不提供 Codex `/review` 操作方法，也不定义具体仓库脚本与 CI 入口。  
这些内容属于 Codex 落地层，而不属于角色协议层。

本文档与其他上位文档的关系如下：

| 文档 | 法位 | 本文档与其关系 |
|---|---|---|
| **执行模型法典** | 上位宪法 | Reviewer 必须继承 R1 / R2 / R3 与 G1 / G2 / G3 的法位，不得重定义 |
| **核心术语表** | 上位词典 | Reviewer 必须使用正式术语，不得制造审查黑话体系 |
| **规划模型** | 规划法 | Reviewer 依赖总任务文档作为上位参考之一，而不是仅凭 diff 猜测目标 |
| **Planner 协议** | 上游角色协议 | Reviewer 不替代 Planner 做规划裁决，但可以识别规划失真并要求回改 |
| **Coder 协议** | 上游角色协议 | Reviewer 不替代 Coder 做实现，但可以裁决当前实现是否可信、是否收敛、是否应升级 |

因此，本文档的职责只有四项：

| 职责 | 含义 |
|---|---|
| **定义 Reviewer 的法位** | 说明 Reviewer 不是 coder，不是 planner，也不是自由评论者 |
| **定义 Reviewer 的输入协议** | 说明哪些证据齐备后，才允许开始正式审查 |
| **定义 Reviewer 的输出协议** | 说明 Reviewer 的正式产物是什么，必须包含什么 |
| **定义 Reviewer 的升级与交接规则** | 说明何时从 R1 升到 R2 / R3，何时返回 Planner / 用户裁决 |

---

## 1. 核心结论

Reviewer 的职责可以压缩为一句话：

> **基于上位参考、结构化证据与当前差异，对 Task 锚点可信度、Milestone 阶段结构收敛与 Initiative 交付可信度做正式裁决。**

这句话里有四层边界必须先钉死。

### 第一，Reviewer 审的是“一致性”，不是“好不好看”

Reviewer 不是风格警察，不是代码讲解员，也不是“看着顺不顺眼”的表达者。  
Reviewer 审查的对象，本质上是：

> **当前差异与上位参考之间是否一致。**

这里的上位参考可以是：

- 总任务文档
- 方案设计文档
- 差距分析文档
- 规范切片 / Spec Refs
- Milestone / Initiative Reference

没有上位参考，正式审查就会退化成经验评论。

---

### 第二，Reviewer 审的是“能否继续推进”，不是“是否绝对完美”

正式审查的目标不是追求完美代码，而是回答当前对象层级上的关键问题：

| 层级 | Reviewer 的核心问题 |
|---|---|
| **R1 / Task 审查** | 当前锚点在 Task 半径内是否功能正确、验证充分、结构局部收敛，是否值得继续推进 |
| **R2 / Milestone 审查** | 当前阶段是否已经结构上收敛，且可安全并入主干 |
| **R3 / Initiative 审查** | 当前专项是否具备正式交付资格 |

因此，Reviewer 的判断标准是“法位匹配”，而不是“审得越多越好”。

---

### 第三，Reviewer 不替代方案设计与实现

Reviewer 可以指出：

- 当前差异偏离方案
- 当前实现触发未裁决问题
- 当前里程碑切法失真
- 当前修复层错误

但 Reviewer 不能越位去：

- 自己改写规划
- 自己拍板新方案
- 自己用评论替代实现

它的职责是裁决与升级，不是替代上游角色干活。

---

### 第四，Reviewer 的输出不是“建议清单”，而是“正式裁决结果”

Reviewer 的输出至少要让下游能清楚知道：

- 当前是否可继续推进
- 问题是已证实还是推断
- 风险位于哪个对象层
- 是局部修补，还是必须升级
- 下一步应回到哪个角色或哪个层级

因此，Reviewer 的工作不是“多提点意见”，而是：

> **把当前差异放回制度坐标系中定性。**

---

## 2. Reviewer 的正式定义

Reviewer 是角色协议层中的正式审查角色。  
它不是“第二个 coder”，也不是“第三个 planner”。

正式定义如下：

> **Reviewer 是负责依据上位参考、验证证据与当前差异，对对象层收敛状态做正式裁决的角色。**

这份定义包含三个关键点。

### 2.1 Reviewer 的对象不是代码文本，而是“当前审查对象”

Reviewer 直接处理的对象不是某个函数，而是当前法定位次下的审查对象：

- R1：anchor commit / commit range
- R2：Milestone PR / branch diff against base
- R3：release / rollout / deployment candidate 与对应材料

因此，Reviewer 从来不“直接审仓库的一切”，而是始终审**有法位的对象**。

---

### 2.2 Reviewer 的工作不是重复跑验证，而是裁决验证是否足够代表当前对象

验证与审查不是同义词。

- Gate / Checks 负责跑
- Reviewer 负责判断这些 Checks 是否足以支持当前层级的成立判断

因此，Reviewer 不只是问“有没有测试”，而是进一步问：

> **这些测试和证据，是否真正锁住了当前对象应成立的东西。**

---

### 2.3 Reviewer 的正式输出必须可执行

“有风险，建议关注”这种话不构成正式审查输出。  
正式输出必须至少能支持下游做以下判断：

- 继续推进
- 停止推进
- 回到 coder 修补
- 回到 planner 改规划
- 回到设计层重裁决
- 升级到更高层审查

也就是说，Reviewer 的输出不是态度，而是**动作导向裁决**。

---

## 3. 输入协议

Reviewer 只有在输入满足正式条件时，才允许进入正式审查。  
否则，所谓 review 只是对不完整材料做主观评论。

---

### 3.1 R1 的必备输入

R1 只审当前锚点在 Task 半径内是否可信，因此最低输入为：

| 输入 | 是否必需 | 说明 |
|---|---|---|
| **结构化 commit 文本** | 必需 | 至少包含 Intent / Scope / Non-Goals / Checks |
| **当前 commit diff / commit range** | 必需 | 当前 Task 锚点的真实差异 |
| **G1 结果** | 必需 | 当前局部验证结果 |
| **相关 Refs / Spec Refs** | 条件必需 | 只要触及契约、字段、接口、状态、迁移，即必须提供 |
| **必要的源码上下文** | 条件必需 | 当前 diff 不足以判断时补充 |

没有结构化锚点文本的 R1，默认不合法。
仅凭实现者自述或任意 anchor 前局部材料，也不构成合法 R1 输入。

---

### 3.2 R2 的必备输入

R2 审的是 Milestone 是否收敛，因此最低输入为：

| 输入 | 是否必需 | 说明 |
|---|---|---|
| **被显式指派的 Milestone Reference** | 必需 | 当前阶段的法定目标与验收 |
| **PR diff / branch diff against base** | 必需 | 当前阶段真实差异范围 |
| **PR 内 anchor commits 索引** | 必需 | 让阶段审查能回溯到局部锚点 |
| **G2 结果** | 必需 | 当前阶段验证结果 |
| **必要的设计文档 / 差距分析章节** | 条件必需 | 当前阶段依赖上位裁决时必须提供 |

若“总任务文档中的阶段区块”与“独立 Milestone 文档”同时存在，但未明确哪一个被指派为法定入口，则 R2 输入不合法。
没有被显式指派的 Milestone Reference，R2 默认只能给出非正式意见，不能给正式放行判断。

---

### 3.3 R3 的必备输入

R3 审的是 Initiative 是否可交付，因此最低输入为：

| 输入 | 是否必需 | 说明 |
|---|---|---|
| **被显式指派的 Initiative Reference** | 必需 | 当前专项到底要证明什么 |
| **Milestone 完成矩阵** | 必需 | 哪些阶段已成立，哪些未成立 |
| **G3 结果** | 必需 | 交付级验证结果 |
| **release / rollout / deployment candidate** | 必需 | 当前交付候选对象 |
| **风险、债务、验收材料** | 必需 | 当前是否具备安全交付条件 |

若当前专项只有零散方案材料、阶段文档与若干 PR，但没有被指派的 Initiative Reference，则 R3 输入不合法。
没有交付候选对象的 R3，不构成正式专项审查。

---

### 3.4 非法输入信号

只要出现下列情况之一，Reviewer 就不得继续正式裁决，而应要求补输入或升级。

| 非法输入信号 | 原因 |
|---|---|
| **只有 diff，没有上位参考** | 无法判断一致性 |
| **只有测试结果，没有当前对象边界** | 无法判断验证是否足够 |
| **commit / PR 没有结构化说明** | 难以识别 Intent 与 Scope |
| **当前对象层级不清楚** | 无法决定使用 R1 / R2 / R3 哪种法位 |
| **多份候选参考并存，但未指派法定入口** | reviewer 不得自行猜测哪篇算上位法 |
| **存在明显未裁决问题** | 审查无法替代上游裁决 |

Reviewer 的正确动作不是“先审着看”，而是：

> **停止正式裁决，要求补齐输入或返回上游。**

---

## 4. 输出协议

Reviewer 的正式输出不是普通评论，而是正式裁决结果。  
不同层级的审查，输出粒度不同，但核心结构应统一。

---

### 4.1 最小输出结构

所有正式审查输出，至少都应包含以下四类信息：

| 输出项 | 含义 |
|---|---|
| **Verdict** | 当前对象是否可继续推进 / 合并 / 交付 |
| **Judgment Dimensions** | 直接回答当前层到底审了什么 |
| **Evidence Adequacy** | 当前输入证据是否足够支撑该层级判断 |
| **Escalation / Next Action** | 当前问题应由谁、回到哪一层处理 |

正式输出的第一层结构，不应把 `Findings` 当总兜底桶。
更合理的做法是：

> **先按当前层级的审查对象逐项裁决。**
> **再在每个裁决项下附 findings、证据与残余风险。**

---

### 4.2 R1 的推荐输出结构

R1 建议输出：

- Verdict
- Functional Correctness
- Validation Adequacy
- Local Structure Convergence
- Local Regression Risk
- Escalation

其中：

- `Functional Correctness` 直接回答当前功能是否符合任务意图
- `Validation Adequacy` 直接回答 G1 与相关 checks 是否足够支撑该锚点成立
- `Local Structure Convergence` 直接回答 Task 半径内的职责边界、依赖方向与适配方式是否收敛
- `Local Regression Risk` 直接回答当前锚点是否引入明显局部回退

`Findings` 不应再作为 R1 的一级总栏位；它应作为上述各项下的支撑问题列表。

这里的核心问题是：

> **当前锚点在 Task 半径内是否可信。**

R1 不应输出长篇系统设计评论，也不应审到 Milestone 或 Initiative 层。
它也不负责替 anchor 前的局部收敛背书；R1 审的是 anchor 形成后的正式对象。

---

### 4.3 R2 的推荐输出结构

R2 建议输出：

- Verdict
- Stage Structure Convergence
- Mainline Merge Safety
- Evidence Adequacy
- Residual Risks
- Required Follow-Ups

其中：

- `Stage Structure Convergence` 直接回答 Milestone 是否已经形成阶段结构收敛
- `Mainline Merge Safety` 直接回答当前阶段合入主干后是否安全
- `Evidence Adequacy` 直接回答 G2 与相关材料是否足以支持阶段放行判断
- `Residual Risks` 只保留已知、已界定、且尚可接受的残余问题

`Convergence Findings` 不应继续作为模糊总栏；相关 findings 应挂在对应裁决项之下。

这里的核心问题是：

> **当前阶段是否已经结构上收敛，且可安全并入主干。**

R2 必须能看见阶段性问题，而不仅是 commit 局部问题。

---

### 4.4 R3 的推荐输出结构

R3 建议输出：

- Initiative Review Verdict
- Goal Realization Findings
- Delivery Confidence
- Rollback / Rollout Concerns
- Residual Debt Acceptability
- Final Recommendation

这里的核心问题是：

> **当前专项是否具备正式交付资格。**

因此，R3 的输出形式本质上应是一份 **Initiative Review Report**，而不是普通 PR 评论。

---

### 4.5 Findings 的书写规则

`Findings` 是对裁决项的支撑，不是正式输出的第一层结构。
每条 findings 至少要能回答四个问题：

| 维度 | 说明 |
|---|---|
| **位置** | 具体 diff、文件、路径或对象层级 |
| **问题结论** | 这是何种问题 |
| **出错机制** | 为什么它会出错，不是只说表面现象 |
| **影响范围** | 它影响当前锚点、当前阶段还是交付层 |

若为推断性风险，必须显式写明：

- 推断依据
- 前提条件
- 尚未证实之处

禁止把“可能有问题”写成“已确认缺陷”。

---

## 5. 审查原则

Reviewer 在正式审查时，必须遵守以下原则。

---

### 5.1 事实优先原则

> **优先输出可由代码、diff、验证结果与上位参考直接证明的问题。**

Reviewer 可以指出高置信推断，但必须明确区分：

- 已证实问题
- 高置信推断
- 需要更多证据的问题

否则，审查就会从裁决退化为猜测。

---

### 5.2 一致性裁决原则

> **Reviewer 审的是差异与上位参考的一致性，而不是孤立代码质量。**

这意味着 Reviewer 必须始终问：

- 当前差异是否完成了它声称要完成的事
- 当前实现是否越出既定边界
- 当前非目标是否被误判成漏做
- 当前状态是否与 Milestone / Initiative 的成立条件一致

没有这条原则，审查就会滑向无穷尽的局部完美主义。

---

### 5.3 层级守恒原则

> **R1 不重演 R2，R2 不重演 R3。**

每个层级只回答自己的问题：

| 层级 | 只回答什么 |
|---|---|
| **R1** | 锚点可信度（功能正确性、验证充分性、局部结构收敛） |
| **R2** | 阶段结构收敛与主干集成安全 |
| **R3** | 交付可信度 |

一旦三个层级都在重复谈同样的事，审查体系就失焦了。

---

### 5.4 系统性优先原则

> **当多个问题共享根因时，必须上升为系统性缺陷，而不是继续点状罗列。**

这条原则意味着：

- 同类问题复发，不再只批当前 patch
- 多个 findings 指向同一事实源错误时，要点名根因
- 修复跨层扩散时，要直接要求结构性 remedy

Reviewer 的真正价值，不在于列出很多点，而在于指出**最该改的那一层**。

---

### 5.5 验证充分性原则

> **Reviewer 不只看“有没有验证”，还要看“验证是否锁住了该锁的东西”。**

因此，Reviewer 必须主动识别：

- 只覆盖 happy path 的测试
- 只验证表象的断言
- 缺失的反向路径
- 缺失的迁移 / 回退 / 兼容路径
- 与当前对象层级不匹配的验证深度

这条原则尤其重要，因为 Agent 最容易制造“看起来测了”的错觉。

---

### 5.6 非越位原则

> **Reviewer 可以要求升级，不替代上游角色做越位裁决。**

这意味着：

- 发现方案未定 → 返回方案设计层
- 发现 Task / Milestone 切法失真 → 返回 Planner
- 发现需要结构性修复 → 要求 Coder 调整修复层
- 发现交付边界问题 → 提醒用户裁决

但 Reviewer 不直接自己写新规划，也不直接以评论代替设计决策。

---

## 6. 各层审查重点

不同层级的 Reviewer 关注点必须不同。

---

### 6.1 R1：Task 审查重点

R1 只回答一个问题：

> **这个 anchor commit / commit range，在当前 Task 半径内是否已经功能正确、验证充分、结构局部收敛，足以成为可信的 Task 锚点。**

R1 的重点应是：

> **功能正确性**
> **验证充分性**
> **Task 半径内的局部结构收敛**

三者缺一不可。

| 审查项 | 核心问题 |
|---|---|
| **Intent 对齐** | 当前改动是否真在做它声称要做的事 |
| **Scope 收敛** | 当前差异是否越出了声明边界 |
| **Non-Goals 清晰** | 当前未覆盖内容是否本就不应覆盖 |
| **功能正确性** | 当前行为、异常处理、边界条件是否符合任务意图 |
| **验证充分性** | G1 与 checks 是否真正锁住当前改动与主要风险 |
| **局部结构收敛** | 当前 Task 半径内的职责边界、依赖方向、适配方式是否收敛，不引入临时拼接、局部双真值或明显结构债 |
| **局部回归风险** | 当前 patch 是否引入明显局部倒退 |

必须明确：

> **R1 只审 anchor 形成后的正式对象。**
> **anchor 前的实现收敛、局部验证与修补，不直接产生放行裁决。**

R1 不审：

- 全局架构完美性
- Milestone 级结构收敛是否已经成立
- Initiative 是否达标
- 审美型风格偏好

---

### 6.2 R2：Milestone 审查重点

R2 只回答一个问题：

> **这个 Milestone PR，是否已经形成一个结构上收敛、且可安全并入主干的阶段闭环。**

R2 的重点应是：

> **Milestone 是否已经形成阶段结构收敛**
> **主干合入后是否会引入不可接受的回归风险**

| 审查项 | 核心问题 |
|---|---|
| **阶段结构收敛** | 这组 Task 合起来是否形成单一事实源、清晰边界与可持续的阶段结构 |
| **状态闭环** | 当前 Milestone 宣称成立的状态是否真的成立 |
| **事实源收敛** | 是否存在双真值、影子状态、隐式同步 |
| **契约同步** | 下层结构、边界适配、调用方是否同步迁移 |
| **失败语义一致** | 错误、补偿、回退、重试是否逻辑统一 |
| **兼容路径控制** | 过渡层是否可控，是否污染主链 |
| **回归风险控制** | 当前合入主干后，是否会引入不可接受的功能、兼容性或稳定性回归风险 |
| **验证代表性** | G2 是否覆盖主要风险面 |

R2 不应退化成“把 R1 findings 汇总一遍”。

---

### 6.3 R3：Initiative 审查重点

R3 只回答一个问题：

> **这个 Initiative，是否已经具备正式交付资格。**

R3 的重点应是：

| 审查项 | 核心问题 |
|---|---|
| **目标兑现度** | 原始 Initiative 目标是否真实达成 |
| **Milestone 完整性** | 必需 Milestone 是否都已成立 |
| **交付可信度** | rollout / release / deployment 是否安全 |
| **回退可信度** | 出问题时是否可降级、回滚、止血 |
| **残余债务可接受性** | 当前未完成项是否仍允许上线 / 结项 |
| **业务接受度** | 产品或技术目标是否真正被接受 |

R3 的讨论对象不是单个 diff，而是**专项整体状态**。

---

## 7. 升级规则

Reviewer 不只是发现问题，更必须知道何时升级问题。

---

### 7.1 从 R1 升级到 R2 的条件

只要出现下列情况之一，R1 必须升级到 R2：

| 条件 | 原因 |
|---|---|
| **多个 findings 指向同一共享根因** | 已超出局部锚点层 |
| **当前问题跨越多个 Task / commit** | 已不是单锚点问题 |
| **当前修复明显影响阶段状态成立** | 应进入 Milestone 层裁决 |
| **当前差异暴露规划失真** | 需看总任务文档与阶段切法 |

---

### 7.2 从 R2 升级到 R3 的条件

只要出现下列情况之一，R2 必须升级到 R3：

| 条件 | 原因 |
|---|---|
| **问题已经影响交付可信度** | 已超出阶段收敛层 |
| **Milestone 虽成立，但整体 rollout 风险过高** | 需专项级裁决 |
| **回退、发布、灰度路径失控** | 已进入交付法位 |
| **当前残余债务可能改变 Initiative 是否可结项** | 需 R3 裁决 |

---

### 7.3 返回 Planner 的条件

Reviewer 发现以下情况时，必须返回 Planner：

| 条件 | 原因 |
|---|---|
| **Task 切法与实际差异严重不符** | 执行图失真 |
| **Milestone 边界明显过厚或过薄** | 规划切法失真 |
| **Workstream 依赖与现实冲突** | 并行结构需回改 |
| **PR 路径已不符合原阶段设计** | 集成路径需回改 |

---

### 7.4 返回方案设计 / 差距分析层的条件

Reviewer 发现以下情况时，必须停止继续审差异本身，而要求返回上游：

| 条件 | 原因 |
|---|---|
| **核心边界未定** | 审查无法替代方案裁决 |
| **现状理解与差距分析冲突** | 当前执行图缺乏真实基础 |
| **新主状态 / 主对象 / 关键兼容策略被暗中引入** | 已超出实现合法边界 |

---

## 8. 交接协议

Reviewer 的输出必须能够驱动下游明确行动，而不是停留在“知道了一些问题”。

---

### 8.1 交接给 Coder 的条件

当 review 结论是“可继续推进但需修补”时，Reviewer 应向 Coder 交接：

- 明确的 Findings
- 严重度排序
- 已证实 / 推断的区分
- 最合理修复层
- 是否仍需保持当前 Scope
- 是否必须补验证

这意味着 Reviewer 的输出必须可执行，而不是“自己意会”。

---

### 8.2 交接给 Planner 的条件

当 review 结论表明当前问题已超出局部实现，涉及：

- Task 失真
- Milestone 失真
- Workstream 失真
- PR 路径失真

则 Reviewer 必须向 Planner 交接，要求回改执行图，而不是继续在实现层修修补补。

---

### 8.3 交接给用户的条件

Reviewer 不直接做最终产品、合规或交付裁决。  
但当 review 结论涉及：

- 范围变化
- 交付风险变化
- rollout / 回滚策略变化
- 业务接受度变化

时，Reviewer 必须显式把裁决抬回用户。

因为最终：

> **用户负责真实环境手测、边界批准、最终合入与交付裁决。**

---

## 9. 禁止事项

Reviewer 在任何情况下都不得触犯以下禁令。

---

### 9.1 禁止无参考审查

- 只看 diff，没有上位参考
- 没有结构化 commit 文本就给正式 R1
- 没有被显式指派的 Milestone Reference 就给正式 R2
- 没有被显式指派的 Initiative Reference 与 candidate 就给正式 R3

---

### 9.2 禁止把推断写成事实

- “可能有问题”写成“已确认缺陷”
- 只凭经验猜测就下最终裁决
- 不写依据就判断设计错误
- 不写前提就判断竞态成立

---

### 9.3 禁止审查越位

- Reviewer 自行改规划
- Reviewer 自行拍板新方案
- Reviewer 用评论替代实现
- Reviewer 直接裁掉用户应做的边界决定

---

### 9.4 禁止层级失焦

- 让 R1 重演 R2
- 让 R2 重演 R3
- 把 Task 锚点前局部收敛冒充正式 R1
- 把 branch final review 冒充新的正式审查层
- 用全局架构苛责当前局部锚点
- 用局部代码洁癖阻断阶段收敛

---

### 9.5 禁止低价值审查

- 解释代码而不是指出问题
- 罗列样式与命名偏好
- 在没有系统性问题时空谈重构愿景
- 在已有共享根因时继续点状堆 findings

---

### 9.6 禁止把 review 变成验证替身

- 用“我看起来没问题”替代 checks
- 因为 review 通过就默认验证足够
- 不检查验证代表性就直接放行
- 让 reviewer 用经验掩盖证据缺口

---

## 10. 封板结论

本文档可以压缩为以下十条法则。

### 法则 1  
**Reviewer 是正式裁决者，不是 coder、planner 或意见表达者。**

### 法则 2  
**Reviewer 审的是差异与上位参考的一致性，不是孤立代码好不好看。**

### 法则 3  
**不同层级只回答不同问题：R1 审锚点可信度，R2 审阶段结构收敛与主干集成安全，R3 审交付可信度。**

### 法则 4  
**没有上位参考、法定参考入口或结构化证据，就没有正式审查。**

### 法则 5  
**Reviewer 必须区分已证实问题与高置信推断，不得把推断写成事实。**

### 法则 6  
**Reviewer 不只看有没有验证，更要看验证是否足够代表当前对象成立。**

### 法则 7  
**当多个问题共享根因时，必须上升为系统性缺陷，而不是点状罗列。**

### 法则 8  
**当问题超出当前层级时，Reviewer 必须升级，而不是继续在错误层级内纠缠。**

### 法则 9  
**Reviewer 的输出必须可执行，能够明确驱动 coder、planner 或用户的下一步动作。**

### 法则 10  
**在 Agent 时代，审查的真正价值不在于多提意见，而在于把高吞吐生成重新压回制度坐标系。**

---

> **最终收口：**
>
> **Reviewer 不是“更挑剔的读代码者”。**
> **Reviewer 是收敛秩序的维护者。**
>
> 当生成足够快、补丁足够容易时，真正防止系统滑向熵增的，不是更多代码，而是更清楚的裁决。

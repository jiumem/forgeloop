# Agent 编程项目核心术语表

> **状态**：Sealed / v1
> **定位**：Agent 编程项目的正式术语裁决文档
> **目标**：统一执行模型、规划模型、质量控制、证据体系与角色协议中的高频名词，消除下位文档、PR、审查意见与实现讨论中的语义漂移
> **适用范围**：需要持续集成、阶段收敛、正式审查、发布纪律的软件工程项目
> **不适用范围**：一次性脚本、临时实验、无需主干收敛与发布纪律的松散型代码工作

---

## 0. 文档定位

本文档只回答一个问题：

> **哪些词在本体系中具有正式法位，分别代表什么，不代表什么。**

它不重复展开《Agent 编程项目执行模型法典》的完整结构，也不负责定义具体仓库的目录、脚本、CI 平台、Prompt 文本、Skill 落点或工具选型。

本文档是**术语裁决文档**，不是实施手册。
它的职责是为后续所有下位文档提供统一词典，使讨论、规划、提交、验证、审查与交付始终运行在同一坐标系内。

---

## 1. 术语使用原则

### 1.1 第一原则：术语表服务于法位稳定，而不是服务于词汇丰富

术语表不是收录常见工程黑话的地方。
它只保留那些**不理解就无法正确使用本体系**的核心词汇。

因此，术语表中的词必须满足至少一项条件：

| 条件        | 含义                      |
| --------- | ----------------------- |
| **对象必要性** | 该词对应正式对象、正式动作、正式证据或正式角色 |
| **裁决必要性** | 该词会直接影响验证、审查、合并、发布或结项判断 |
| **协作必要性** | 该词是跨文档、跨 PR、跨审查语境的高频共用词 |

---

### 1.2 第二原则：主名称优先使用自然语言，简称只作辅助

术语表的主任务是降低语义摩擦，而不是强化内部黑话。

因此：

> **正式主名称优先使用自然中文或稳定英文术语。**
> **简称、代号、缩写只作为辅助索引，不作为主词条。**

例如：

* 主名称应写作 **任务门**，而不是直接写作 `G1`
* 主名称应写作 **里程碑审查**，而不是直接写作 `R2`

---

### 1.3 第三原则：内部机制术语不得伪装成第一等世界对象

有些概念在内部实现上极其重要，但并不是所有读者都必须首先掌握的第一等概念。

因此：

> **内部质量控制元结构术语，不进入主术语表正文。**
> **它们应进入附录，作为框架维护与制度解释所用的内部术语。**

这类术语包括但不限于：

* Assertion Plane
* Profile Plane
* Execution Surface
* Gate Family
* Gate Suite
* Gate Catalog
* Formal Entrypoint Contract

---

## 2. 核心执行对象术语

这一组术语定义项目执行世界中的第一等对象。
它们不是任务大小的不同叫法，而是不同法位的正式对象。

| 术语             | 正式定义                                      | 不是什么                         | 典型作用                 |
| -------------- | ----------------------------------------- | ---------------------------- | -------------------- |
| **Initiative** | 一次完整目标闭环的专项对象，可来源于产品、架构、治理、性能、质量、安全或基础设施等 | 不是 Epic 的随意替身；不是某个页面；不是某个 PR | 定义专项目标、交付边界与结项条件     |
| **Milestone**  | 一个阶段状态、能力闭环或不变量组合首次正式成立的收敛边界              | 不是时间节点；不是施工进度条；不是目录分类标签      | 定义阶段成立、PR 收敛与阶段验收坐标  |
| **Task**       | 最小可实现、可验证、可继续推进的工程闭环                      | 不是泛待办；不是任何零散改动；不是“顺手优化一下”    | 定义局部实现动作、锚点提交与局部验证单元 |

---

### 2.1 Initiative

Initiative 是整套执行体系中最高层的项目执行对象。
它的核心含义不是“任务多一点”，而是：

> **一次完整战役。**

它的合法结束条件不是“某个 PR 合并”，而是：

> **目标兑现，具备发布、全量、交付或正式结项资格。**

因此，Initiative 的判断标准始终是**交付成立**，而不是**差异发生**。

同时，Initiative 也是后续执行体系的默认主入口。  
终端用户与正式调度上下文，默认都应围绕 Initiative 推进，而不是把 Task 当作常态主入口。

---

### 2.2 Milestone

Milestone 的本质不是过程，而是状态。
它用于回答：

> **现在是否已经形成一个可以被正式宣布成立的阶段闭环。**

因此，Milestone 是：

* 阶段收敛边界
* 状态成立断点
* PR 集成与阶段验证的法定对象

它不是：

* 甘特图上的月份标签
* 某个功能列表
* 某个目录名
* 某条分支名

---

### 2.3 Task

Task 是最小工程闭环，不是最小粒度的“事”。

一个合法 Task 至少同时具备四项属性：

| 属性       | 含义                 |
| -------- | ------------------ |
| **明确输入** | 依赖哪些规范、契约、接口、前置状态  |
| **明确动作** | 到底改什么，而不是模糊描述      |
| **明确输出** | 代码、测试、迁移、脚本、文档或配置等 |
| **明确边界** | 哪些内容不在本任务范围内       |

因此，Task 对应的是**局部收敛责任**，而不是主干收敛责任。

它也是**内部最小执行原子**，而不是终端用户默认操作对象。

---

## 3. 收敛动作术语

这一组术语定义状态如何被保存、同步、集成与交付。
它们不是对象本体，而是对象收敛到物理世界时的正式动作。

| 术语             | 正式定义                   | 不是什么                         | 典型作用                 |
| -------------- | ---------------------- | ---------------------------- | -------------------- |
| **Commit**     | 为差异打锚点的 Git 记录对象       | 不是任务完成证明                     | 为验证、审查、回滚与归因提供精确差异定位 |
| **Push**       | 将当前分支状态同步为远端快照的动作      | 不是阶段成立信号；不是正式放行动作            | 触发预警校验、形成恢复点、供并行工作接续 |
| **PR**         | Milestone 进入主干前的正式收敛容器 | 不是 Initiative 本体；不是随意的差异堆叠容器 | 承载阶段性集成、验证、审查与修补     |
| **Release**    | 面向正式交付对象的版本发布动作        | 不是单纯打 tag；不是任意发包             | 作为 Initiative 交付动作之一 |
| **Flag**       | 以开关或灰度方式控制能力暴露的交付动作    | 不是配置噪音；不是临时绕过机制              | 承载灰度、回退、逐步放量与发布控制    |
| **Deployment** | 将候选产物部署到目标环境的交付动作      | 不是状态成立本身                     | 将已验证候选物推进到运行环境       |

---

### 3.1 Commit

Commit 的正式职责不是表示“做完了”，而是：

> **为差异提供精确锚点。**

它服务于四类下游动作：

* 验证
* 审查
* 回滚
* 归因

因此，Commit 是证据对象，不是成就对象。

---

### 3.2 Push

Push 的法位是**同步动作**，不是**准入动作**。

它的价值主要在于：

1. 形成远端恢复点
2. 支持多 worktree / 多 Agent / 多环境接续
3. 触发低成本预警校验与漂移发现

因此：

> **Push 承担同步与预警语义，不承担放行语义。**

---

### 3.3 PR

PR 的正式法位应始终理解为：

> **Milestone 收敛容器。**

它不是最终交付对象，也不是长期专项本体，更不是单次快照。
一个打开中的 PR 可以持续 push、持续 review、持续修补，直到该 Milestone 满足合并条件。

---

## 4. 规划辅助术语

这一组术语不属于三层执行对象主骨架，但它们构成 Planner 的正式工作语言。
如果缺失这些词，下位规划文档、任务拆分与实施讨论会持续漂移。

| 术语                      | 正式定义                | 不是什么            | 典型作用                      |
| ----------------------- | ------------------- | --------------- | ------------------------- |
| **Requirement**         | 需求输入对象，定义问题来源与目标动机  | 不是实现方案；不是页面清单   | 作为规划起点                    |
| **Scope**               | 当前轮次的覆盖边界           | 不是对象层级；不是实现细节全集 | 定义本轮工作边界                  |
| **Non-Goals**           | 当前轮次明确不做的内容         | 不是“以后再说”的空话     | 压缩范围，防止误判漏做               |
| **Workstream**          | 可相对独立并行推进的作战线       | 不是对象主层；不是目录名    | 支持并行拆线与 Agent 分工          |
| **Dependency**          | 前后置依赖与约束关系          | 不是任务本体          | 构建 PR 序列与执行 DAG           |
| **Acceptance Criteria** | 成立标准与验收条件           | 不是愿景性口号         | 定义 Milestone / Task 的成立依据 |
| **Residual Risk**       | 当前阶段明确接受、但尚未消除的残余风险 | 不是遗漏项掩护语        | 为审查、发布与后续治理提供依据           |

---

### 4.1 Requirement

Requirement 的重点不在于“做什么功能”，而在于：

> **为什么这件事值得进入执行体系。**

它负责表达问题与价值，不负责提前决定解决方案形态。

---

### 4.2 Scope 与 Non-Goals

这两个词必须成对出现。
只有 Scope 而没有 Non-Goals，边界一定会膨胀；只有 Non-Goals 而没有 Scope，执行对象一定会漂移。

因此：

> **Scope 决定本轮覆盖什么，Non-Goals 决定本轮刻意不覆盖什么。**

---

### 4.3 Workstream

Workstream 是并行结构对象，不是主骨架对象。
它用于回答：

> **同一个 Initiative 或 Milestone 内，哪些作战线可以相对独立推进。**

因此，它应服务于**并行拆解**，而不是重复发明执行层级。

---

## 5. 质量控制术语

这一组术语定义验证与审查体系中的正式质量控制对象。
主名称优先使用自然语言；内部正式名与简称作为辅助说明出现，不反客为主。

---

### 5.1 验证类术语

| 主名称      | 内部正式名                   | 简称     | 正式定义                                                    |
| -------- | ----------------------- | ------ | ------------------------------------------------------- |
| **任务门**  | Task Gate Profile       | G1     | Task 级 blocking 验证，用于证明当前锚点已具备继续推进的最低工程可信度              |
| **里程碑门** | Milestone Gate Profile  | G2     | Milestone 级 blocking 验证，用于证明一组 Task 已形成可安全进入主干的阶段闭环     |
| **专项门**  | Initiative Gate Profile | G3     | Initiative 级 blocking 验证，用于证明累计 Milestone 已满足发布、全量或结项条件 |
| **预警校验** | Shadow Check Profile    | Shadow | 非阻断预警验证，用于尽早暴露未来会在里程碑门或专项门爆炸的问题                         |

---

### 5.2 审查类术语

| 主名称       | 内部正式名                     | 简称 | 正式定义                                                     |
| --------- | ------------------------- | -- | -------------------------------------------------------- |
| **任务审查**  | Task Review Profile       | R1 | 针对 anchor commit 或 commit range 的正式审查，用于判断当前 Task 锚点是否可信 |
| **里程碑审查** | Milestone Review Profile  | R2 | 针对 Milestone PR 的正式审查，用于判断阶段状态是否真正收敛                     |
| **专项审查**  | Initiative Review Profile | R3 | 针对 Initiative 交付候选的正式审查，用于判断专项是否具备正式交付资格                 |
| **预警审查**  | Shadow Review Profile     | SR | 非阻断预警审查，用于在正式里程碑审查或专项审查之前尽早发现语义漂移与范围膨胀                   |

---

### 5.3 质量控制术语的使用规则

#### 规则 A

**正式文档优先使用主名称。**

例如优先写：

* 任务门失败
* 里程碑审查结论
* 预警校验红灯

而不是一上来只写：

* G1 fail
* R2 blocked
* Shadow red

---

#### 规则 B

**简称仅用于表格、图示、配置索引、口头速记与内部缩略表达。**

简称的存在是为了减轻重复书写成本，而不是为了制造黑话壁垒。

---

#### 规则 C

**“编排”不作为术语表主名称。**

虽然这些对象在内部法位上属于 `Profile`，但术语表面向读者时，不直接暴露 L2 机制词。
因此：

> 主名称写作 **任务门**，而不是 **任务门编排**。
> 内部正式名保留为 **Task Gate Profile**。

---

#### 规则 D

**Shadow 不作为正式中文主名称。**

在质量控制体系中，中文主名称统一使用：

> **预警校验 / 预警审查**

`Shadow` 仅作为内部英文正式名与简称索引使用。
`影子校验 / 影子审查` 不作为正式中文主名称。

---

### 5.4 Task 内部检查术语

以下术语描述 **Task 锚点形成前的局部收口机制**。
它们属于 Task 内部检查层，不进入正式 Gate / Review 法位，也不新增第四层正式质量控制对象。

| 术语 | 正式定义 | 不是什么 | 典型作用 |
| --- | --- | --- | --- |
| **规范符合性检查** | Task 内部、anchor 前的非正式局部检查，用于确认当前实现是否命中 Task / Spec-Refs 所要求的边界与内容 | 不是 R1；不是 Milestone 级审查 | 为当前 Task 的局部修补与继续实现提供反馈 |
| **代码质量检查** | 在规范符合性检查通过后进行的 Task 内部、anchor 前的非正式局部检查，用于暴露明显工程质量、可维护性与局部回归风险 | 不是 R1 / R2；不是 branch 级正式审查 | 帮助当前 Task 在进入 anchor 前完成局部清理与降噪 |

因此：

> **Task 可以先做内部检查，再沉淀 anchor commit。**
> **但只有 anchor commit + G1 + R1，才构成正式 Task 收口。**

---

## 6. 证据与参考术语

这一组术语定义执行、验证、审查与交付所依赖的正式证据对象。
它们不属于执行对象主骨架，但它们决定整个系统是否可证明、可回滚、可裁决。

| 术语                           | 正式定义                         | 不是什么               | 典型作用                       |
| ---------------------------- | ---------------------------- | ------------------ | -------------------------- |
| **Anchor Commit**            | Task 正式差异锚点提交                | 不是任意 commit；不是 WIP | 作为任务门与任务审查的正式输入            |
| **Fixup Commit**             | 针对 gate / review 失败而做的正式修补提交 | 不是随手补丁；不是噪音提交      | 为修补提供可审查、可归因的正式差异          |
| **Branch Checkpoint**        | push 后形成的分支快照输入对象            | 不是阶段成立证据           | 作为预警校验与预警审查的输入             |
| **Spec Slice**               | Task 级规范切片                   | 不是整篇设计文档强制全读       | 作为触及契约、字段、接口、状态、迁移时的局部上位参考 |
| **Spec-Refs**                | 规范切片索引字段                     | 不是正文装饰             | 显式声明 Task 所引用的法定规范来源       |
| **Milestone Reference Doc**  | 里程碑级正式上位参考文档                 | 不是可有可无的补充说明        | 作为里程碑审查的法定依据               |
| **Initiative Reference Doc** | 专项级正式上位参考文档                  | 不是项目愿景宣讲稿          | 作为专项审查的法定依据                |
| **Release Candidate**        | 发布候选对象                       | 不是已正式发布版本          | 作为专项门与专项审查输入物之一            |
| **Rollout Candidate**        | 灰度或放量候选对象                    | 不是正式全量状态           | 作为灰度放量与交付验证输入              |
| **Deployment Candidate**     | 部署候选对象                       | 不是部署完成事实           | 作为交付与环境推进前的验证输入            |

---

### 6.1 Anchor Commit

Anchor Commit 是 Task 进入正式质量控制体系的起点。
它的存在意味着：

> **当前差异已经收口到一个可验证、可审查、可定位的任务锚点。**

因此，没有 anchor commit，就不应进入正式任务审查。

---

### 6.2 Spec Slice 与 Spec-Refs

Task 审查不默认读取整篇设计文档。
只有在触及契约、字段、接口、状态或迁移时，才必须附带相关规范切片。

因此：

> **Spec Slice 是局部法源。**
> **Spec-Refs 是显式索引，不允许 reviewer 靠猜。**

---

### 6.3 Reference Doc

Milestone 与 Initiative 都必须拥有正式上位参考文档。
否则，审查就会退化成对 diff 的孤立评论，而非对**差异与上位法一致性**的裁决。

---

## 7. 执行角色术语

这一组术语定义执行体系中的正式角色。
它们不是世界对象，而是承担规划、实现与审查职责的角色协议承载者。

| 术语           | 正式定义                        | 不是什么            | 典型作用                   |
| ------------ | --------------------------- | --------------- | ---------------------- |
| **Planner**  | 负责将 Requirement 压缩为可执行结构的角色 | 不是秘书型任务拆分器      | 定义范围、里程碑、作战线、依赖与 PR 序列 |
| **Coder**    | 负责在既有边界内交付正确、收敛、可验证实现的角色    | 不是机械翻译需求的代码生成器  | 实施改动、维护边界、补齐验证、暴露残余风险  |
| **Reviewer** | 负责对锚点、阶段与交付进行正式审查的角色        | 不是纯风格警察；不是代码讲解员 | 发现裂痕、判断收敛、升级系统性风险      |

---

### 7.1 Planner

Planner 的正式职责不是列 Todo，而是：

> **把模糊需求压成可并行、可验收、可合入的执行图。**

因此，它必须掌握：

* Requirement
* Scope / Non-Goals
* Workstream
* Dependency
* Acceptance Criteria
* Initiative / Milestone / Task / PR 序列

---

### 7.2 Coder

Coder 的法位是**实现责任承担者**。
它必须在既有边界内完成正确收敛，而不是通过局部补丁制造新的双真值、影子状态或半迁移逻辑。

---

### 7.3 Reviewer

Reviewer 的法位不是“提出建议的人”，而是**正式裁决者**。
它必须区分：

* 已证实问题
* 高置信推断
* 范围外议题
* 必须升级的系统性裂缝

---

### 7.4 关于 Architect 的裁决

**Architect 不作为独立一级角色进入本术语表。**

原因不是架构工作不重要，而是当前体系中，架构性职责已被以下两部分吸收：

* Planner 的架构性规划职责
* Reviewer 的架构性裁决职责

因此，默认不单列 Architect。
若某项目需要专项化的架构设计或架构审查协议，应在下位文档中以专项协议方式出现，而不回灌为核心术语。

---

## 8. 非核心、非默认与禁止混用术语

这一组术语用于明确边界：哪些词不是本体系的正式主词，哪些用法被明令禁止。

### 8.1 非核心术语

| 术语          | 裁决                                            |
| ----------- | --------------------------------------------- |
| **Epic**    | 不进入本体系主术语表；若项目内部需要，可在下位文档中映射，但不得覆盖 Initiative |
| **Feature** | 仅作为产品能力描述使用，不作为执行对象术语                         |
| **Story**   | 不进入本体系主术语表                                    |
| **Subtask** | 不作为默认核心对象，避免与 Task 法位冲突                       |
| **Final Review** | 不作为默认核心对象层术语；若发生 branch-wide review，应归入 R2 的辅助输入或预警层，而不是新增第四层正式审查 |

---

### 8.2 禁止混用语义

| 禁止说法                  | 原因                         |
| --------------------- | -------------------------- |
| 把 Branch 说成 Milestone | Branch 只是连续工作上下文容器，不是对象层级  |
| 把 PR 说成 Initiative    | PR 是 Milestone 收敛容器，不是专项本体 |
| 把 Push 说成阶段成立证据       | Push 只承担同步与预警语义            |
| 把 Commit 说成任务完成证明     | Commit 只是差异锚点，不是完成裁决       |
| 把 Task 内部检查说成 R1      | 内部检查只是 anchor 前局部收口，不是正式任务审查 |
| 把 branch final review 说成第四层正式审查 | branch 级 final review 只能作为 R2 辅助输入或预警，不产生新法位 |
| 用 G1 / R2 直接替代主名称     | 简称不能取代正式术语                 |
| 把内部元结构术语当作第一等概念       | 会造成法位混乱与理解门槛上升             |

---

## 9. 术语之间的最小关系图

整套术语可压缩为下面这条主链：

> **Requirement 进入 Planner 语义世界，被压缩为 Initiative。**
> **Initiative 按 Milestone 分段收敛。**
> **Milestone 由一组 Task 推动成立。**
> **Task 可先经过内部检查，再通过 Anchor Commit 进入任务门与任务审查。**
> **Milestone 通过 PR 进入里程碑门与里程碑审查。**
> **Initiative 通过 Release / Rollout / Deployment Candidate 进入专项门与专项审查。**

这条主链之外，Push 与 Branch Checkpoint 提供预警输入；Spec Slice、Reference Doc 与 Candidate 对象提供法定证据。

---

## 10. 附录：质量控制内部术语

本附录术语重要，但不进入主术语表正文第一层法位。

| 术语                             | 正式定义   | 作用                                                   |
| ------------------------------ | ------ | ---------------------------------------------------- |
| **Assertion Plane**            | 断言层    | 定义 Gate / Review 所依赖的正式断言对象与断言结构                     |
| **Profile Plane**              | 配置层    | 定义阻断/非阻断质量控制对象如何被组合、挂接与触发                            |
| **Execution Surface**          | 执行面    | 将质量控制对象映射到脚本、CLI、CI、PR checks、release pipeline 与外部目标 |
| **Gate Family**                | 门族     | 表示同一类验证门的上位集合                                        |
| **Gate Suite**                 | 门套件    | 一组协同执行的门组合                                           |
| **Gate Catalog**               | 门目录    | 断言与门配置的正式目录化资产                                       |
| **Formal Entrypoint Contract** | 正式入口契约 | 约束验证或审查从何处进入执行面                                      |

---

## 11. 封板结论

本文档将整套体系中最容易漂移的名词正式裁决为六组主术语与一组附录术语：

| 组别           | 内容                                                                                                                                            |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| **核心执行对象术语** | Initiative / Milestone / Task                                                                                                                 |
| **收敛动作术语**   | Commit / Push / PR / Release / Flag / Deployment                                                                                              |
| **规划辅助术语**   | Requirement / Scope / Non-Goals / Workstream / Dependency / Acceptance Criteria / Residual Risk                                               |
| **质量控制术语**   | 任务门 / 里程碑门 / 专项门 / 预警校验 / 任务审查 / 里程碑审查 / 专项审查 / 预警审查                                                                                          |
| **证据与参考术语**  | Anchor Commit / Fixup Commit / Branch Checkpoint / Spec Slice / Spec-Refs / Milestone Reference Doc / Initiative Reference Doc / Candidate 对象 |
| **执行角色术语**   | Planner / Coder / Reviewer                                                                                                                    |
| **附录术语**     | Assertion Plane / Profile Plane / Execution Surface 等内部元结构名词                                                                                  |

---

> **最终收口：**
>
> **术语不是修辞。**
> **术语是执行世界的边界。**
>
> 只有当对象、动作、证据、质量控制与角色协议拥有稳定命名时，Agent 时代的软件工程才可能在高吞吐生成之上维持长期收敛。

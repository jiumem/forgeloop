# Forgeloop 4.0.0

Forgeloop 4.0.0 收敛了从 Spec 修订、系统设计、Ticket 审查到执行验收的整条工作流。核心变化是让自动化保持可控：Reviewer 必须受正式合同约束，执行最多拥有一次自动纠偏机会，最终验收只封存已经产生且可行动的证据，不再在交付末端制造没有后续处理路径的新问题。

## Planning Revision 原位维护

- `to-tickets` 发现父 Spec 合同缺口时，返回同一 Spec 的 Planning Revision 入口，不再废弃旧 Spec Issue 并创建替代品；
- `to-spec` 在用户确认后原位发布新的有效 Revision，保留旧 Revision 的历史关系和原生回读证据；
- 完整 Ticket 正文只维护在同一份草案文档中，会话只展示 Ticket 摘要和本轮变更，避免重复大段正文；
- 同一高风险 Ticket 的 Design Reviewer 跨修订轮次复用；只有原 Reviewer 不可用、失去只读独立性或审查权威发生变化时才替换；
- 第三次连续 non-PASS 后强制执行尺度审查，把新增事实源、状态、生命周期和失败保证逐项映射回父 Spec、ADR、Delivery Acceptance 或 Cross-seam Invariant，删除 Reviewer 擅自扩大的要求。

## 正式系统设计文档

- 不新增专用 Skill，也不把可变的模块级设计塞入 ADR；
- `grill-with-docs` 判断多个模块、Ticket、实现会话或 Reviewer 是否必须共享同一组接口、权限、状态、事务、恢复或兼容决定，并创建或原位修订正式 Design Document；
- `to-spec` 保存稳定文档引用，Spec 与 ADR 继续拥有产品范围和长期架构决定；
- `to-tickets` 只在共享实现决定确实缺失时返回 `DESIGN_DOCUMENT_REQUIRED`，不会因为功能“大”或 `HIGH_RISK` 就机械要求文档，也不会把同一设计复制到每张 Ticket。

## 有界自动纠偏

- 每个 Ticket 最多两个修复周期，每个周期最多三轮实际改变候选代码或测试的普通修复；
- Cycle 1 耗尽后先持久化 `RUN_PAUSED / REPAIR_BUDGET`，再由 fresh Correction Coder 做只读 Exhaustion Diagnosis；
- 只有可信、可证伪、仍在 Scope 内且会收敛设计的 `AUTO_REPAIR_RENEWAL` 才自动进入唯一 Cycle 2，不需要中断进度向用户确认；
- Cycle 2 刷新 Coder 和两名 Ticket Reviewer，并在周期内复用；Cycle 2 耗尽后停止自动修改，不创建 Cycle 3、replacement Ticket 或 replacement Run；
- Scheduler 只校验诊断是否完整且与原生事实矛盾，不代替 Coder 判断技术机制，也不重写 Reviewer Findings；
- `NO_REPAIR` 只适用于缺少正式权威或越界的 Spec-axis Finding，Standards Reviewer 的职责和路由保持独立。

## Final Integration Gate 与 Acceptance Seal

- `SHARED` Spec 的 Final Integration Gate 是最后一个允许产生实现 Finding 的阶段；它验证固定交付范围和必要的真实端到端旅程，并把可行动 Finding 交给 `to-tickets`；
- Final Acceptance 不再创建 Reviewer、不运行新的语义验收、不产生 Finding 或 repair key；
- Scheduler 只从已有 Ticket Verdict、Integration Result、Final Gate evidence、有效 Revision、membership 和最终目标 Commit 判断 Seal Eligibility；
- `ACCEPTANCE_RESULT` 继续作为耐久封存记录，通过字面量安全发布和精确原生回读形成 Acceptance Seal；
- 多 Spec Initiative 的成员结果必须绑定同一最终 Commit，Initiative Seal 确认后才关闭成员 Specs、父 Initiative 并释放 Claim；
- Seal 输入缺失时返回现有事实责任主体和恢复动作，不再制造没有对应执行环节的末端审查结果。

## 兼容性与升级边界

- 4.0.0 不新增 Event、Tracker Item 类型、持久化状态机或 parser，也不改写历史 Spec、Ticket、Run 和 Checkpoint；
- 行为合同发生不兼容变化：取消 Acceptance Repair、限制为最多两个修复周期，并要求 Planning Revision 原位维护；
- 不要在同一个活动 Run 内混用 3.6.1 与 4.0.0。现有运行和旧 Acceptance Repair 记录的处理方式见[迁移指南](../migrations/3.6.1-to-4.0.0.md)；
- 20 个正式 Skill、11 个用户入口和 9 个模型可调用能力保持不变。

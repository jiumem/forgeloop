# 契约阻塞只设置一次原子裁决门禁

当执行发现必须改变 Spec、Scope、Acceptance 或 ADR 时，内部 Design Review 必须先自行收敛并通过，之后才能向用户提交包含契约差异、Ticket 调和与 Run 恢复动作的完整方案并请求一次批准。Reviewer 发现的非契约性设计缺口由 Agent 自主修订，不得反复请求用户确认；只有出现此前未展示且确实改变用户所有契约决定的新事实时，才允许重新进入用户裁决，以兼顾执行自动化与契约治理权。用户明确批准“调和并继续”后，该批准同时授权契约发布、Open Ticket 调和、原生回读验证与原 Run 恢复，不再要求分别显式调用对应工作流；用户明确要求暂不继续时，Run 才在调和完成后保持暂停。

## 后果

- `CONTRACT_BLOCKER` 不授权 Coder、Reviewer 或 `run-initiative` 直接修改正式契约。
- 用户批准发生在内部设计门禁通过之后，并覆盖已展示的完整调和事务。
- 批准后的实现细化不使批准失效；真正改变已批准契约的差异必须重新裁决。
- 调和事务任一步失败都不得发布 `RUN_RESUMED`；修复执行故障后从正式 Tracker 事实幂等续跑，不重复询问已经确认的契约决定。
